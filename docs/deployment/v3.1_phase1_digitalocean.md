## Visual Detector Stack

The visual extraction pipeline now uses:

- **Detector:** `RFDETRLarge` from the `rfdetr` library
- **Tracker:** `ByteTrackTracker` from Roboflow `trackers`
- **Decode:** CPU-based frame decoding via OpenCV (keeps GPU lane free for detection)

Key rules:

- CUDA is required for detection — the worker will fail fast if no GPU is available
- No silent CPU fallback for the detector
- No silent model downgrade (do not swap RFDETRLarge for Medium or Small)
- FP16 inference is the default (`pytorch_cuda_fp16`)
- TensorRT FP16 is the optimized path (`tensorrt_fp16`) — fully implemented
- When `tensorrt_fp16` is set, the detector auto-exports ONNX and builds a `.engine` on first run, then caches it
- Engines are GPU-specific: delete the cache and re-run when switching GPU families
- Detection is batched for throughput; tracking remains sequential in frame order

Detection thresholds are tuned for tracking quality, not raw speed. ByteTrack benefits from lower-confidence detections being available for association.

## Scheduling Decision

The current intended Phase 1 schedule is:

1. normalize/download media
2. start **pyannote cloud** immediately
3. start the **local GPU worker**
4. inside the worker, run local tasks in serial:
   - visual extraction first (RF-DETR Large + ByteTrack on GPU)
   - `emotion2vec+` second
   - `YAMNet` third
5. join the outputs
6. build Phase 1 artifacts
7. continue into Phases 2-4

Why:

- pyannote cloud overlaps the longest local work
- local GPU contention stays simple and predictable
- visual extraction starts first because it is expected to dominate wall-clock time
- `emotion2vec+` and `YAMNet` stay on the same GPU lane instead of contending in parallel

## Benchmarking Requirements

Before making TensorRT the production-default, benchmark both backends on the target DO GPU:

1. `CLYPT_PHASE1_VISUAL_BACKEND=pytorch_cuda_fp16` — baseline
2. `CLYPT_PHASE1_VISUAL_BACKEND=tensorrt_fp16` — optimized path

The first TensorRT run will export ONNX and build the `.engine` (one-time cost, typically 2-5 minutes). Subsequent runs reuse the cached engine.

Collect:

- wall-clock video processing time
- mean detector latency (`tracking_metrics.mean_detector_latency_ms`)
- end-to-end effective FPS (`tracking_metrics.effective_fps`)
- VRAM peak
- detection count parity (same or higher detection count)
- track continuity quality (same or better track lengths)

Success condition:

- TensorRT preserves or improves practical detection/tracking quality vs PyTorch CUDA
- TensorRT is faster in effective FPS
- both paths are faster than the old YOLO path on the target worker
