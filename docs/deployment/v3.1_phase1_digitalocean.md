# V3.1 DigitalOcean Phase 1 Deployment

**Status:** Live — native GPU path re-validated on 2026-04-06  
**Branch:** `v3.1-refactor`  
**Droplet:** `clypt-v3-phase1-worker` — latest successful ASR validation on `gpu-h200x1-141gb`

This document is the V3.1 replacement for the old main-branch DigitalOcean Phase 1 deployment notes. It reflects the actual bring-up experience and is the authoritative operational reference for the refactored worker.

## Scope

This deployment path runs:

- Phase 1 local worker tasks on a GPU droplet via a parent coordinator + branch workers:
  - `visual` branch: RF-DETR Large + ByteTrack on GPU
  - `audio` branch: VibeVoice-ASR (default: native ~7B subprocess, Gradio parity; optional HF ~9B in-process) + NeMo Forced Aligner + `emotion2vec+`
  - `yamnet` branch: YAMNet on CPU in V1
- the refactored Phase 1 sidecar build, then Phases 2–4

Pyannote cloud has been removed. All audio processing now runs locally on the GPU droplet.

It does **not** revive the old LR-ASD-era worker stack.

## Visual Detector Stack

- **Detector:** `RFDETRLarge` from the `rfdetr` library
- **Tracker:** `ByteTrackTracker` from Roboflow `trackers`
- **Decode:** CPU-based frame decoding via OpenCV (keeps GPU lane free for detection)

Key rules:

- CUDA is required for detection — the worker will fail fast if no GPU is available
- No silent CPU fallback for the detector
- No silent model downgrade (do not swap RFDETRLarge for Medium or Small)
- FP16 inference is the default (`pytorch_cuda_fp16`)
- TensorRT FP16 is the optimized path (`tensorrt_fp16`) — fully implemented
- When `tensorrt_fp16` is set, the detector auto-exports ONNX and builds a `.engine` on first run, then caches it
- Engines are GPU-specific: delete the cache and re-run when switching GPU families
- Detection is batched for throughput; tracking remains sequential in frame order

## Scheduling

Phase 1 is now a parent-coordinated runtime with three child processes:

1. Download + audio extract
2. Upload **video** to GCS (audio stays local)
3. Parent coordinator launches:
   - `visual` branch on GPU
   - `audio` branch on GPU
   - `yamnet` branch on CPU
4. Coordinator waits for branch `result.json` files, joins them into Phase 1 artifacts, then continues into Phases 2–4

Why this shape:

- keeps the two GPU-heavy lanes explicit: `visual` and `audio`
- keeps YAMNet off the GPU in V1 to avoid adding a third GPU branch
- preserves process isolation between RF-DETR and the validated native VibeVoice subprocess stack
- keeps failure recovery straightforward: the coordinator owns child lifecycle and terminates siblings on hard failure

### Visual pipeline

```bash
CLYPT_PHASE1_VISUAL_BACKEND=pytorch_cuda_fp16
CLYPT_PHASE1_VISUAL_BATCH_SIZE=4
CLYPT_PHASE1_VISUAL_THRESHOLD=0.35
CLYPT_PHASE1_VISUAL_SHAPE=640
CLYPT_PHASE1_VISUAL_TRACKER=bytetrack
CLYPT_PHASE1_VISUAL_TRACKER_BUFFER=30
CLYPT_PHASE1_VISUAL_TRACKER_MATCH_THRESH=0.7
CLYPT_PHASE1_VISUAL_DECODE=cpu
```

**Critical:** `CLYPT_PHASE1_VISUAL_SHAPE` must be `640`. RF-DETR's DINOv2 backbone requires inputs divisible by 32, and the model's patch size is 16 — `560` triggers an `AssertionError` at runtime.

**Critical:** `CLYPT_PHASE1_VISUAL_TRACKER_MATCH_THRESH` must be `0.7`, which is ByteTrack's upstream default for `track_activation_threshold`. Setting it higher causes tracks to not activate until after several frames of confident detection, losing the beginning of person appearances.

## Known Gotchas from the First Real Run

These are things that were not obvious from docs alone. Record them here so they don't get rediscovered.

### YouTube format selection

yt-dlp's Python API does **not** auto-discover the JS runtime used for the EJS n-challenge solver. If `CLYPT_YOUTUBE_JS_RUNTIMES` is not set to the bun runtime, yt-dlp will fail with "Requested format is not available" even when cookies are valid.

### GCS bucket ACL vs uniform access

Our GCS bucket (`clypt-storage-v3`) has uniform bucket-level access enabled. The `blob.make_public()` fallback path fails with a `400 BadRequest`. The only working path is V4 signed URLs via a service account key. Service account key is mandatory.

### VibeVoice model download (Hugging Face cache)

On first run, the active model (7B native or 9B HF) downloads to `~/.cache/huggingface/`. This can add several minutes on a fresh droplet. Subsequent runs use the cache. If the droplet is destroyed, the cache is lost — pre-warm by running a one-line import in the **same** interpreter that will load the model (native venv for `VIBEVOICE_BACKEND=native`, main venv for `hf`).

### `flash-attn` wheel ABI mismatch on the native venv

The prebuilt `flash-attn` wheel did not load cleanly against the droplet’s `torch 2.6.0+cu124`; import failed with an undefined symbol error. The successful fix was:

1. install CUDA toolkit 12.4 on the host so `nvcc` matches PyTorch’s CUDA build
2. rebuild `flash-attn==2.8.3` from source inside `/opt/clypt-phase1/.venv-vibevoice-native`
3. verify `import flash_attn, flash_attn_2_cuda` before switching back to `VIBEVOICE_ATTN_IMPLEMENTATION=flash_attention_2`

Without this, the safe fallback is `sdpa`.

### Parallel runtime contract on the droplet

The startup contract enforced by `factory.py` is:

- `CLYPT_PHASE1_PARALLEL_ENABLED=1`
- `CLYPT_PHASE1_PARALLEL_GPU_BRANCH_LIMIT=2`
- `CLYPT_PHASE1_YAMNET_BRANCH_DEVICE=cpu`

If the GPU branch limit is changed or YAMNet is pointed at GPU, the worker fails fast at startup instead of silently drifting away from the validated V1 runtime.

### Branch failures now have per-branch logs

When Phase 1 fails, inspect these in order:

1. the top-level worker log from `nohup`
2. `metadata/branch_summary.json` inside the run workspace
3. the specific `branches/<branch>/branch.log` referenced by the summary or failure payload

The branch summary is the fastest way to tell whether the failure happened during branch launch, branch execution, or coordinator-side result validation.

### `wave` reported a bogus ~6.8 hour duration

`joeroganxflagrant.wav` exposed bad `wave` metadata (`nframes = 2**30 - 1`) even though `torchaudio.info(...)` reported the correct ~13.1 minute duration. We now probe duration with `torchaudio` only and fail fast if that metadata is invalid.

### Stereo HF input must be downmixed to mono

The HF VibeVoice path rejected dual-channel WAV input with `Audio should be mono`. The fix was to average channels to mono before resampling and inference. The native subprocess path handles the production droplet case; the stereo downmix remains important for the local HF fallback.

### HF VibeVoice path requires transformers>=5.3.0

The in-process 9B path uses `VibeVoiceAsrForConditionalGeneration`. Confirm with `pip show transformers` in the main venv when using `VIBEVOICE_BACKEND=hf`. The native subprocess does not use the main venv’s `transformers` for inference.

### RF-DETR COCO person class ID is 1, not 0

COCO class 0 is background/reserved in RF-DETR's output. Person is class 1. Filtering on class 0 returns zero detections. Check `rfdetr_detector.py` `COCO_PERSON_CLASS_ID = 1`.

### ByteTrackTracker ignores its config if not passed explicitly

The `ByteTrackTracker` constructor signature is `ByteTrackTracker(lost_track_buffer, frame_rate, track_activation_threshold)`. Calling it with no arguments silently uses defaults (`frame_rate=30`, `track_activation_threshold=0.25`) regardless of what you set in `VisualPipelineConfig`. Always pass all three explicitly. See `tracker_runtime.py`.

### ByteTrackTracker frame rate must match the actual video FPS

The `lost_track_buffer` is in frames, and its timeout is `lost_track_buffer / frame_rate` seconds. If you pass `frame_rate=30` for a 23.98 fps video, the timeout is silently wrong. We now detect actual FPS via `cv2.VideoCapture` and pass it at tracker initialization. See `visual.py`.

### `sv.Detections` boolean indexing fails with non-uniform `data` fields

RF-DETR's `sv.Detections` output may contain `data` dict fields (like `class_name`) with shapes that don't match the primary detection array. Filtering via `det[person_mask]` triggers an `IndexError`. The fix is to reconstruct `sv.Detections` manually without carrying `det.data` through the filter. See `rfdetr_detector.py`.

### The cookies file is not synced by rsync

`.env.local` and the YouTube cookies file both live outside the repo directory on the droplet. They must be uploaded manually via `scp` after each new droplet provisioning.

## Benchmarking (Pending)

Before making TensorRT the production-default, benchmark both backends on the target DO GPU:

1. `CLYPT_PHASE1_VISUAL_BACKEND=pytorch_cuda_fp16` — baseline (currently validated at ~89 fps on H100)
2. `CLYPT_PHASE1_VISUAL_BACKEND=tensorrt_fp16` — optimized path (ONNX export + engine build on first run)

Collect:

- wall-clock video processing time
- mean detector latency (`tracking_metrics.mean_detector_latency_ms`)
- end-to-end effective FPS (`tracking_metrics.effective_fps`)
- VRAM peak
- detection count parity
- track continuity quality

Success condition: TensorRT preserves or improves detection/tracking quality vs PyTorch CUDA, and is faster in effective FPS.
