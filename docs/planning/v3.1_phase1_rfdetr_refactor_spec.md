# V3.1 Phase 1 RF-DETR Refactor Spec

**Status:** Implemented  
**Branch:** `v3.1-refactor`  
**Scope:** Phase 1 visual extraction only  
**Out of scope:** Phases 2-4 semantics, pyannote cloud, `emotion2vec+` schema, `YAMNet` schema, participation timeline, camera intent

## Goal

Replace the current Ultralytics YOLO-based person detector in the V3.1 Phase 1 worker with `RFDETRLarge`, while keeping a separate tracker layer and preserving the existing Phase 1 visual artifact contracts:

- `shot_changes`
- `tracks`
- `person_detections`
- `tracking_metrics`

This refactor is intentionally bounded:

- `RFDETRLarge` becomes the detector
- a separate tracker remains responsible for persistent IDs
- Phase 1 artifact schemas remain stable
- Phases 2-4 should not need to change because of this detector swap

## Product Decision

Use:

- `RFDETRLarge` for person detection
- a separate tracking layer after detection
- the same downstream shot-local `tracks` and `person_detections` artifacts

Do not:

- collapse detection and tracking into one opaque stack
- change the semantic pipeline because of this detector refactor
- revive LR-ASD, face identity, or old binding logic

## Why This Refactor

The current V3.1 visual runtime uses:

- `YOLO("yolo11n.pt")`
- Ultralytics `model.track(...)`
- ByteTrack through the Ultralytics integration

This has two problems:

1. Detector choice is still tied to the old YOLO stack.
2. Tracking is embedded inside the detector runtime rather than being a clean explicit stage.

Moving to `RFDETRLarge` plus a separate tracker gives us:

- a cleaner detector/tracker boundary
- a detector that is no longer tied to Ultralytics runtime behavior
- a more stable long-term Phase 1 visual architecture

## Architecture

### New visual pipeline

The new Phase 1 visual path should be:

1. probe video metadata
2. detect shot boundaries
3. decode frames
4. run `RFDETRLarge` person detection on GPU
5. convert detections into `supervision.Detections`
6. run a separate tracker stage on those detections
7. normalize tracker output into the existing `tracks` schema
8. split tracks at shot boundaries
9. build `person_detections`
10. emit `tracking_metrics`

### Detector/tracker boundary

Detector responsibilities:

- detect persons only
- emit per-frame boxes, confidences, and class IDs
- remain stateless across frames

Tracker responsibilities:

- assign persistent IDs across frames
- update tracks from frame-level detections
- remain replaceable without changing detector logic

Post-processing responsibilities:

- normalize the tracker output into current artifact shape
- split global tracks into shot-local tracks
- generate `person_detections`

## Detector Choice

Use:

- `RFDETRLarge`

Reason:

- this is the user-selected detector
- official RF-DETR docs list `RFDETRLarge` as the large detection checkpoint
- official latency numbers are reported with TensorRT FP16 at batch size 1, which is aligned with a GPU-optimized deployment path

Important:

- do not switch to smaller RF-DETR sizes for speed by default
- do not reduce detection resolution below the modelâ€™s expected operating regime just to improve speed
- do not replace `RFDETRLarge` with `Medium` or `Small` unless the user explicitly asks for that tradeoff later

## Tracker Choice

Use:

- `ByteTrackTracker` from Roboflow `trackers`

Reason:

- official Roboflow trackers docs explicitly show RF-DETR + ByteTrack together
- ByteTrack is designed for real-time MOT and low fragmentation
- it preserves the â€śdetector first, tracker secondâ€ť architecture we want
- it keeps the refactor bounded because our current artifacts already assume track IDs

Do not:

- depend on a detector-integrated tracking mode
- put tracking responsibility back inside the detector library

## Artifact Contract Stability

The following artifacts must remain semantically stable:

### `shot_changes`

No change in schema.

The shot detector remains independent from the detector swap.

### `tracks`

No schema change.

Still emit rows with:

- `frame_idx`
- `track_id`
- bbox coordinates
- confidence
- normalized geometry
- shot-local track identifiers after splitting

### `person_detections`

No schema change.

Still derived from shot-split tracks.

### `tracking_metrics`

Keep existing fields where they still make sense, but update backend labels to reflect RF-DETR + ByteTrack rather than Ultralytics tracking.

Example:

- replace `tracker_backend: ultralytics_yolo11_bytetrack`
- with something like `tracker_backend: rfdetr_large_bytetrack`

## CUDA / GPU Optimization Rules

These are mandatory implementation rules for the new worker.

### Hard requirements

- The Phase 1 visual path must run on CUDA, not CPU.
- If CUDA is unavailable on the worker, fail fast.
- Do not silently fall back to CPU for detection.
- Do not silently fall back to a smaller model.
- Do not silently lower the model input shape to chase speed.

### Worker scheduling

The intended Phase 1 schedule stays:

1. start pyannote cloud in parallel
2. run visual extraction on the worker GPU
3. run `emotion2vec+` on the same worker GPU
4. run `YAMNet` on the same worker GPU
5. join outputs

These local tasks stay serial on the same GPU lane.

Do not:

- run visual extraction, `emotion2vec+`, and `YAMNet` concurrently on one GPU by default
- create GPU contention just to increase theoretical parallelism

### RF-DETR inference optimization rules

Use all of the following unless a specific hardware benchmark disproves one:

- `torch.inference_mode()` around inference
- `model.eval()`
- explicit CUDA device placement
- FP16 / half-precision inference
- model warmup before timed processing
- fixed-shape inference where possible
- batched frame inference where the model and VRAM allow it
- pinned host memory for CPU-to-GPU transfer if we build a batched loader
- `cudnn.benchmark = True` when frame shape is stable

### RF-DETR-specific optimization rules

The official RF-DETR API warns that inference latency is higher unless `model.optimize_for_inference()` is called.

So the implementation should:

1. load `RFDETRLarge`
2. call `optimize_for_inference(...)`
3. use FP16 dtype
4. benchmark the chosen batch size on the target DO GPU

Important:

- use FP16 optimization, not INT8, for the default path
- do not introduce quantization-based accuracy tradeoffs in the default worker

### TensorRT rule

The worker should be built with a TensorRT-ready path for RF-DETR.

Default optimization target:

- `RFDETRLarge`
- TensorRT-capable export/deployment path
- FP16

Implementation rule:

- if TensorRT export/runtime for `RFDETRLarge` reaches correctness parity and better throughput on the target DO GPU, that should become the default inference backend
- otherwise keep a pure PyTorch CUDA fallback

This means the runtime should support two RF-DETR backends:

1. `pytorch_cuda_fp16`
2. `tensorrt_fp16`

But only one should be active per run.

### No â€śspeed by degradationâ€ť shortcuts

The following are disallowed as default optimizations:

- frame skipping
- lower detector size
- lower detector family
- higher confidence threshold just to cut detections
- INT8 / aggressive quantization
- lower input shape than the chosen model regime
- CPU fallback

These may only be introduced later as explicit operational tuning modes, not as hidden defaults.

## Frame Processing Strategy

### Decode path

Prefer:

- CPU video decode
- frame queue feeding GPU detection batches

Reason:

- video decode does not need to consume the same GPU lane as detection
- the GPU should spend time on RF-DETR inference, not media parsing

### Batch strategy

The visual worker should support micro-batching of frames for detection.

Recommended first implementation:

- configurable frame batch size
- benchmarked per GPU type
- default chosen empirically for the DO GPU target

Important:

- batching is for detector throughput
- tracking remains logically sequential in frame order

So the intended pattern is:

1. decode next N frames
2. run batched RF-DETR detection
3. feed detections back into ByteTrack in chronological order

## Thresholding And Class Filtering

### Class filtering

Person-only detection remains the default.

That means:

- filter to the person class only
- do not emit generic COCO detections into our Phase 1 visual ledgers

### Confidence threshold

Use a tuned threshold, but do not over-tighten it for speed.

Reason:

- ByteTrack benefits from lower-score detections being available for association
- overly aggressive thresholding hurts track continuity

Implementation rule:

- detector confidence threshold must be tuned for tracking quality, not just raw detector speed

Responsibility:

- own ByteTrackTracker lifecycle

Responsibility:

- metadata probe
- shot detection
- detector stage
- tracker stage
- artifact post-processing

## Benchmarking Requirements

Before making the new worker path the default, benchmark on the target DO GPU:

1. RF-DETR PyTorch CUDA FP16
2. RF-DETR TensorRT FP16

Collect:

- wall-clock video processing time
- mean detector latency
- end-to-end effective FPS
- VRAM peak
- detection count parity
- track continuity quality

Success condition:

- new path preserves or improves practical detection/tracking quality
- and is faster than the current YOLO path on the target worker

## Final Recommendation

Proceed with a bounded detector refactor:

- `RFDETRLarge` replaces YOLO for person detection
- `ByteTrackTracker` remains a separate tracker stage
- visual artifacts stay stable
- the worker is explicitly CUDA-first, FP16, and TensorRT-ready
