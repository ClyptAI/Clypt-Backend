# Clypt V3.1 Refactor Spec

**Status:** Draft v1  
**Date:** 2026-04-03  

## TL;DR

This refactor should make Clypt V3.1 a two-lane system:

- lane 1: semantic understanding, graph construction, and clip candidate ranking
- lane 2: human-guided speaker participation grounding and post-candidate camera direction

The graph is not only an internal clip-generation mechanism.
It is also a user-facing acceleration surface in the Cortex graph UI:

- AI can suggest promising nodes, clusters, and candidate clips
- comments, retention, and trend signals can point the user toward likely high-value regions
- the creator/editor should still be able to think like a human and assemble or refine clips faster using that semantic map

Core decisions locked in for this spec:

- remove LR-ASD entirely
- keep diarized speaker IDs as the global audio identity backbone
- keep shot-local tracklets as the visual identity backbone
- use raw diarized speaker turns as the immutable atomic ledger and merged semantic units as the graph node primitive
- assemble clip candidates from local groups of adjacent / connected nodes rather than creating clips directly from fixed windows
- separate `speaker participation` from `camera intent`
- always build semantics, graph structure, and clip candidates before participation grounding and camera direction
- Use a dedicated speech emotion/prosody model and a dedicated audio event model instead of relying on Gemini to infer all tone/audio signals

## Goals

- improve semantic and grammatical clip boundaries
- reduce cost and latency compared with broad multimodal reasoning on every segment
- preserve Clypt's graph-first value:
  - semantic relationships across moments
  - audience resonance reasoning
  - trend relevance reasoning
- preserve the product ethos of "breaking down the video for the user"
- accelerate human editing, not just autonomous clip generation
- make the Cortex graph UI a first-class editorial surface where creators can see what semantics, comments, retention, and trends are pointing at
- support explicit user-directed rerendering for follow, reaction, split, and wide layouts

## Non-Goals

- preserving compatibility with the current Phase 2A-5 implementation
- preserving LR-ASD or any audiovisual active-speaker model path
- building the future supervised training loop in MVP
- building fully open-ended keyframed camera direction in MVP

## Hard Constraints

- no LR-ASD at all
- fixed windows must not be the graph node primitive
- speaker participation and camera direction must remain separate artifacts
- speaker participation and camera intent is always post-candidate-selection

## Phase Map

V3.1 should be read as a six-phase system:

1. `Phase 1: Timeline Foundation`
   Build the canonical timeline, visual shot/tracklet index, and audio sidecar timelines (speech emotion + non-speech sounds).
2. `Phase 2: Node Construction, Classification, And Embedding`
   Build merged semantic-unit nodes from local turn neighborhoods, reconcile batch edges, and compute durable node embeddings.
3. `Phase 3: Graph Construction`
   Build the semantic graph from merged semantic-unit nodes plus their relationships.
4. `Phase 4: Candidate Retrieval, Subgraph Selection, And Ranking`
   Retrieve promising nodes, build local subgraphs, and use Gemini to choose and rank candidate clips.
5. `Phase 5: Finalist Participation Grounding And Camera Intent`
   Once candidate clips exist, collect participation labels and camera direction for the shortlisted finalists (still 16:9 here).
6. `Phase 6: Render Planning And Render`
   Compile render instructions from editorial grounding on finalists and render the 9:16 output clips.

## Phase 1: Timeline Foundation

Build the full aligned foundation:

- canonical transcript timeline
- shot list
- shot-local tracklet index
- speech emotion / prosody timeline
- audio event timeline

Primary outputs:

- `canonical_timeline.json`
- `speech_emotion_timeline.json`
- `audio_event_timeline.json`
- `shot_tracklet_index.json`
- `tracklet_geometry.json`

### 1. Canonical Transcript Timeline

This is the primary alignment backbone for downstream semantics.

It should contain:

- pyannote word-level transcription with timestamps
- diarized speaker turns (with voiceprints if they exist)

For V3.1, the canonical transcript backbone should come from the full pyannote stack via API, including Precision-2 diarization and pyannote STT orchestration.
That means the primary transcript and diarization source should be pyannote rather than a split transcription-plus-diarization design.

Planned pyannote usage:

- default path: `POST /v1/diarize` with `model=precision-2`
- set `transcription: true` with `parakeet-tdt-0.6b-v3` to request speaker-attributed transcription outputs
- overlapping diarization as the default source of truth for speaker-turn structure; `exclusive` is NOT set to true
- identification path: along with all of the above, hit `POST /v1/identify` when stored recurring voiceprints are available for likely speakers in the run
  - this endpoint does diarization and voiceprinting but no transcription, so the voiceprinting from /identify will have to be merged with output from /diarize

Selection rule:

- if no recurring voiceprints are available for the run, use `diarize`
- if recurring voiceprints are available, use `diarize` + `identify` and merge

Important:

- voiceprints support speaker identification only
- they should be treated as a labeling accelerator and bootstrap signal
- they should not replace user review or the participation-grounding workflow

Relevant pyannote outputs for V3.1:

- `diarization`
- `wordLevelTranscription`
- `turnLevelTranscription`
- `identification`
- `voiceprints`

`turnLevelTranscription` should be treated as a readable convenience layer.
It is useful as punctuated readable text for UI and model context.

Explicit non-goals for V3.1:

- do not compute a separate sentence-boundary artifact in the initial version
- do not compute a separate clause-boundary artifact in the initial version

### 2. Visual Shot / Tracklet Index

This is the primary visual structure layer.

It should contain:

- shot boundaries
- shot-local tracklets
- lightweight tracklet descriptors (human-friendly; cheap geometry and UI metadata only)
- tracklet geometry per frame (renderable motion data)

Descriptor fields:

- `tracklet_id`
- `shot_id`
- `start_ms`
- `end_ms`
- `representative_thumbnail_uris`

Not part of the MVP descriptor set:

- face-presence
- face-visible fraction
- face-quality signals
- dominant_colors

Non-goal:

- global face identity computation is not required for speaker truth in V3.1 because diarization already provides the global audio identity backbone

### 3. Speech Emotion / Prosody Timeline

This should be generated by a dedicated speech emotion model such as `emotion2vec+`, run on raw pyannote speaker turns.

V3.1 should treat this artifact as a turn-level speech-emotion sidecar:

- the canonical timeline still owns the transcript and turn boundaries
- `emotion2vec+` adds scored emotional evidence for each turn
- merged semantic graph nodes in Phase 2 should inherit and aggregate the emotion evidence of their constituent turns rather than redefining the emotion segmentation layer

The stored output should follow the model's native 9-class label set:

- `angry`
- `disgusted`
- `fearful`
- `happy`
- `neutral`
- `other`
- `sad`
- `surprised`
- `unknown`

For each turn-sized segment, store:

- the primary emotion label
- the primary emotion score
- the full per-class score distribution

This gives Clypt a stable low-level answer to:

- how the speech is being delivered (emotionally flat, heightened, tense, surprised, or otherwise notable)
- where tone shifts occur

without forcing Gemini to infer all of that from transcript text alone.

### 4. Audio Event Timeline

This should be generated by a dedicated audio event model: `YAMNet`.

`YAMNet` is a general audio-event classifier, but V3.1 should use it as a non-speech curated reaction / crowd / music-event layer rather than as a generic 521-class dump.

Planned V3.1 curation policy using official YAMNet class names:

- Primary Classes:
  - `Whispering`
  - `Laughter`
  - `Giggle`
  - `Snicker`
  - `Belly laugh`
  - `Chuckle, chortle`
  - `Crying, sobbing`
  - `Whimper`
  - `Wail, moan`
  - `Sigh`
  - `Groan`
  - `Grunt`
  - `Gasp`
  - `Shout`
  - `Bellow`
  - `Whoop`
  - `Yell`
  - `Screaming`
  - `Clapping`
  - `Cheering`
  - `Applause`
  - `Chatter`
  - `Crowd`
  - `Hubbub, speech noise, speech babble`
  - `Cough`
  - `Throat clearing`
  - `Sneeze`
  - `Sniff`
- Secondary Music Classes:
  - `Music`
  - `Background music`
  - `Theme music`
  - `Soundtrack music`
  - `Sad music`
  - `Tender music`
  - `Exciting music`
  - `Scary music`
- Ternary classes:
  - `Singing`
  - `Choir`
  - `Chant`
  - `Rapping`
  - `Humming`
  - `Beatboxing`
  - `Pop music`
  - `Hip hop music`
  - `Rock music`
  - `Heavy metal`
  - `Punk rock`
  - `Progressive rock`

Ignore by default in V3.1:

- generic speech classes such as `Speech`, `Conversation`, and `Narration, monologue`
- most environmental / animal / machine / household / weather classes
- genre-only music classes unless later graph or clip-selection logic proves them useful

Normalization policy:

- do not persist raw YAMNet patch outputs (it scans audio in short overlapping sub-second windows) as product artifacts
- convert patch-level detections into merged intervals in `audio_event_timeline.json`
- use class-specific thresholds rather than one global threshold
- require short contiguous support for noisy classes unless a single patch is very high confidence
- if two or more different kept classes are simultaneously active and each passes its own threshold, keep all of them

## Phase 2: Discourse Unit Construction, Classification, And Embedding

Phase 2 should be broken into four sub-stages.

#### Phase 2A: Raw Turn Neighborhood Construction

Phase 2A should build local Gemini input batches from the canonical timeline while keeping raw diarized turns as the only legal boundary markers.

Start from:
- pyannote speaker turns

Planned V3.1 rule:

- raw diarized speaker turns are the immutable atomic ledger, not the final graph nodes
- do not split a speaker turn under any circumstances
- do not pre-merge turns deterministically before Gemini reviews them
- merged semantic units may only be formed by combining contiguous whole turns
- do not use fixed windows as the graph node primitive

Each raw turn should carry:

- start / end timestamps
- word IDs
- `turn_id`
- speech emotion events overlapping the turn
- audio events overlapping the turn

Build overlapping local neighborhoods for Gemini merge review:

- each batch should contain a contiguous target block of turns
- each batch should also include a small left / right halo for context, such as the previous `2-3` turns and next `2-3` turns
- the halos are context only; only the interior target block is authoritative for that batch
- if a run has 80 raw turns, it should be processed in a manageable number of contiguous batch calls rather than one call per turn

These raw turns remain the source of truth behind all later merged nodes.

#### Phase 2B: Gemini Merged-Unit Construction And Classification

Gemini generative calls should happen here.

In V3.1, the same local Gemini batch call should do both:

- decide which contiguous whole turns belong in one merged semantic unit
- assign the final canonical node classification and flags for each merged unit proposed inside the target block

This coupling is intentional:

- the merge decision depends on what kind of semantic unit the model believes it is seeing
- the node type depends on whether the surrounding turns belong together as one complete thought or interaction
- separating merge from semantic classification too early would make the merge logic brittle and under-informed

Gemini's job in Phase 2B:

- partition the target turns into merged semantic units (~7 - 20s), IF NEEDED (some may not need merging)
- only cut on existing raw turn boundaries
- never invent timestamps
- never skip turns
- use the halo only as context for deciding where the target-block unit boundaries should land
- assign exactly one primary node type per merged semantic unit from the concrete MVP taxonomy:
  - `claim`
  - `explanation`
  - `example`
  - `anecdote`
  - `reaction_beat`
  - `qa_exchange`
  - `challenge_exchange`
  - `setup_payoff`
  - `reveal`
  - `transition`
- assign zero or more node flags:
  - `topic_pivot`
  - `callback_candidate`
  - `high_resonance_candidate`
  - `backchannel_dense`
  - `interruption_heavy`
  - `overlap_heavy`
  - `resumed_topic`
- summarize each merged semantic unit in a graph-friendly way

Each merged semantic unit should carry:

- start / end timestamps
- `source_turn_ids`
- word IDs
- overlapping speech emotion evidence aggregated from its constituent turns
- overlapping audio-event evidence aggregated from its constituent turns
- node type
- node flags
- node summary

Important classification boundary:

- comment, trend, and retention indicators are not node types and not node flags
- those must remain separate overlays that can later be attached to one or more nodes without changing the core semantic node taxonomy

#### Phase 2C: Edge-Batch Reconciliation

Because the local Gemini merge/classification batches overlap, the nodes created near batch edges must be reconciled before they become canonical graph nodes.

V3.1 should use a second, smaller Gemini batch call for this purpose:

- collect only the edge nodes from adjacent overlapping batches
- ask Gemini whether overlapping edge proposals should be:
  - merged/modified to be one semantic unit
  - kept as two distinct neighboring semantic units
- keep this reconciliation call narrow and local; it is not another full merge pass

After this reconciliation step, the resulting merged units become the final canonical semantic graph nodes.

#### Phase 2D: Durable Node Embeddings

Once the merged semantic-unit nodes exist and have been reconciled, compute durable embeddings for each node.

These embeddings support:

- retrieval to find anchor nodes for top clips 
- retrieval to find audience-liked nodes
- retrieval to find trend-relevant nodes
- later video/graph search for the creator

Primary output:

- `semantic_graph_nodes.json`

Optional debug/checkpoint outputs:

- `turn_neighborhoods_debug.json`
- `merge_debug.json`
- `classification_debug.json`

## Phase 3: Graph Construction

Build the graph from the merged semantic-unit nodes created in Phase 2.

Important:

- nodes should remain local, chronological semantic units built from contiguous whole turns
- global semantic relationships should create edges, not force distant units into one merged node
- the graph should preserve both local interaction structure and long-range thematic continuity

The concrete MVP node taxonomy remains intentionally simple:

- each node is a merged semantic unit built from one or more contiguous raw turns
- each node carries one primary `node_type` and supporting node flags chosen in Phase 2
- comment / trend / retention information stays outside the primitive node taxonomy as overlays or tags

Phase 3 should be broken into four sub-stages.

#### Phase 3A: Deterministic Structural Edge Construction

The purely structural edges should be built without Gemini:

- `next_turn`
  - chronological adjacency: the target node is the next merged semantic unit in timeline order
- `prev_turn`
  - chronological adjacency in the reverse direction: the target node is the immediately previous merged semantic unit
- `overlaps_with`
  - temporal overlap: both merged units are active over overlapping time ranges

These should be derived directly from node ordering and timestamps.

#### Phase 3B: Gemini Local Semantic Edge Batches

Use Gemini for the local rhetorical / conversational edges:

- `answers`
  - direct response: the target node functionally answers the earlier node
- `challenges`
  - pushback or dispute directed at an earlier node without necessarily being a flat contradiction
- `contradicts`
  - meaning-level conflict: the target node clashes with or negates the earlier node's claim or framing
- `supports`
  - reinforcing evidence or agreement: the target node backs up the earlier node's claim or framing
- `elaborates`
  - extension or deepening: the target node adds detail, nuance, or expansion to an earlier node
- `setup_for`
  - rhetorical setup: one node establishes the setup, tension, premise, or expectation for a later node
- `payoff_of`
  - rhetorical payoff: one node resolves, lands, or pays off an earlier setup node
- `reaction_to`
  - emotional or conversational reaction: one node is primarily a response beat to an earlier node
- `escalates`
  - intensity ramp: the target node increases stakes, force, extremity, or intensity relative to the earlier node without resolving it

Use overlapping merged-node neighborhoods:

- each Gemini call should contain a contiguous target block of merged nodes
- include a small halo on both sides for local context
- Gemini may use halo nodes for reasoning, but should emit authoritative edges only for the target block

For each returned edge, Gemini should provide:

- `source_node_id`
- `target_node_id`
- `edge_type`
- short `rationale`
- `confidence` score from 0 to 1

#### Phase 3C: Long-Range Edge Adjudication

Use a second, narrower Gemini pass only for:

- `callback_to`
  - deliberate return/reference: a later node intentionally invokes an earlier moment rather than merely sharing a topic
- `topic_recurrence`
  - organic topic resurfacing: the same theme or topic returns later after intervening material

Do not use prompts for this.
Do not run full all-pairs comparison.

Instead:

1. every merged node already has an embedding from Phase 2
2. for each node, retrieve the top `k` semantically closest later nodes
3. treat those as candidate long-range pairs
4. send only those candidate pairs to Gemini for adjudication

Important semantic direction:

- the retrieval/search step runs from earlier node to likely later matches
- if Gemini confirms a true callback or recurrence, the edge itself usually points from the later node back to the earlier node

For each adjudicated long-range edge, Gemini should provide:

- `source_node_id`
- `target_node_id`
- `edge_type`
- short `rationale`
- `confidence` score from 0 to 1

#### Phase 3D: Reconciliation And Dedupe

Gemini proposes semantic edges.
The pipeline canonicalizes, dedupes, and resolves conflicts deterministically.

##### Exact dedupe

If multiple batches produce the exact same:

- `source_node_id`
- `target_node_id`
- `edge_type`

collapse them into one canonical edge and retain useful metadata such as:

- `support_count`
- supporting `batch_ids`
- average `confidence`, if present

##### Local batch conflict handling

If overlapping local batches produce different semantic edge types for the same source/target pair:

- allow multiple edges only when they are compatible
- otherwise keep one canonical edge

Examples of usually compatible combinations:

- `supports` + `elaborates`
- `reaction_to` + `payoff_of`
- `setup_for` + `escalates`

Examples of usually incompatible combinations:

- `answers` vs `challenges`
- `supports` vs `contradicts`
- `setup_for` vs `payoff_of` in the same direction on the same pair

##### Long-range pair dedupe

For long-range edges:

- exact duplicate pair/type proposals should collapse the same way as local edges
- if both `callback_to` and `topic_recurrence` are proposed for the same later-earlier pair, prefer `callback_to` when the later node explicitly refers back; otherwise keep `topic_recurrence`

##### Minimal deterministic resolution algorithm

For each unique source/target node pair:

1. gather all proposed semantic edges for that pair
2. group them by `edge_type`
3. compute `support_count` for each edge type
4. keep all compatible edge types with enough support
5. for incompatible edge types, choose the winner by:
   - higher `support_count`
   - then higher `confidence`, if confidence is present
   - then a fixed specificity / precedence rule when still tied

Recommended tie-breaking intuition:

- prefer more specific relations over weaker generic ones
- for example, prefer `answers` over `elaborates` when the response is clearly answering
- prefer `contradicts` over `challenges` when the meaning-level clash is explicit
- prefer `callback_to` over `topic_recurrence` when the later node deliberately points back to the earlier one

Structural edges from Phase 3A do not participate in Gemini conflict resolution.
They are computed once deterministically.

Primary outputs:
- `semantic_graph_edges.json`

## Phase 4: Candidate Retrieval, Subgraph Selection, And Ranking

Phase 4 should no longer assume that the best clip already exists as a pre-merged semantic span.

Instead, it should:

1. retrieve promising seed nodes
2. build local subgraphs around those seeds using deterministic graph expansion
3. send each local subgraph to Gemini so it can choose strong clip candidates within that subgraph
4. send the pooled candidate set to Gemini for final filtering and ranking

Do not optimize for renderability risk in MVP.
The user will always provide participation grounding and camera intent for finalists at minimum.

Seed-node retrieval sources may include:

- duration-scaled no-prompt meta-prompts for what kinds of moments make strong clips in this video
- comment-cluster prompts
- retention-cluster prompts
- explicit trend prompts
- explicit user query prompts

Duration-scaled no-prompt meta-prompt count:

- under `5 minutes`: `2` meta-prompts
- `5 to 15 minutes`: `4` meta-prompts
- `15 to 25 minutes`: `6` meta-prompts
- `25 to 60 minutes`: `8` meta-prompts
- over `60 minutes`: `10` meta-prompts

Optional later tuning:

- prompt count may be downshifted when node density is unusually low
- exact node-density thresholds are intentionally deferred until real run data is available

Planned retrieval and subgraph-construction rule:

- embed each query or meta-prompt using the same embedding model used for durable node embeddings in Phase 2D
- retrieve the top `5` matching seed nodes per prompt by cosine similarity against durable node embeddings
- build local subgraphs with weighted expansion scoring over eligible local graph neighbors
- use strong temporal locality and limited hop depth as the default guardrails
- cap each local subgraph by:
  - maximum duration
  - maximum node count
  - maximum branching factor
- do not build subgraphs from arbitrary distant thematic matches even if those nodes share a topic

Recommended default builder caps:

- maximum duration: `45s`
- maximum node count: `10`
- maximum hop depth: `2`
- maximum branching factor: `3`
- minimum normal expansion score: `3.0`

Recommended expansion score shape:

- `expansion_score = edge_weight + time_bonus + closure_bonus + adjacency_bonus - hop_penalty - branch_penalty`
- evaluate candidate neighbors using the strongest connecting edge from the current subgraph
- add the highest-scoring eligible neighbors first
- stop normal expansion when:
  - no candidate clears the minimum score
  - maximum duration is hit
  - maximum node count is hit
  - maximum branching factor is hit
  - hop depth is exhausted

Recommended V3.1 edge weights for subgraph growth:

- `setup_for = 4.0`
- `payoff_of = 4.0`
- `reaction_to = 3.5`
- `answers = 3.5`
- `contradicts = 3.5`
- `challenges = 3.0`
- `escalates = 3.0`
- `supports = 2.5`
- `elaborates = 2.5`
- `next_turn = 1.5`
- `prev_turn = 1.5`
- `overlaps_with = 1.0`
- `callback_to = 0.75`
- `topic_recurrence = 0.5`

Recommended time bonus:

- overlap or boundary gap `<= 1.5s`: `+2.0`
- gap `<= 4s`: `+1.0`
- gap `<= 8s`: `+0.5`
- gap `<= 15s`: `+0.0`
- gap `> 15s`: ineligible

Recommended closure bonus:

- candidate is `payoff_of` an included `setup_payoff` node: `+1.5`
- candidate is `answers` an included `qa_exchange` node: `+1.5`
- candidate is `reaction_to` an included `claim`, `reveal`, or `setup_payoff` node: `+1.0`
- candidate is `contradicts` or `challenges` a strong included `claim`, `explanation`, or `challenge_exchange` node: `+1.0`

Recommended adjacency bonus:

- immediate `next_turn` / `prev_turn` neighbor: `+0.5`
- fills a local chronological gap between already-included nearby turns: `+0.5`

Recommended penalties:

- hop penalty:
  - first hop: `0`
  - second hop: `-0.75`
- branch penalty:
  - no penalty for the first strong branch on a side
  - `-0.5` for additional branching pressure beyond that

Gemini's job in Phase 4:

- receive one local subgraph centered on a promising seed
- choose the best self-contained clip candidates that exist within that subgraph
- return for each candidate:
  - included node IDs
  - clip start / end timestamps
  - score
  - short rationale
- return up to `3` clips from a subgraph, but only when all returned clips are actually strong and meaningfully distinct
- return fewer than `3` clips, including zero, when the subgraph does not justify more

Recommended Gemini input payload for subgraph review:

- each reviewed subgraph should be serialized in a compact, deterministic order
- recommended node payload per included node:
  - `node_id`
  - `start_ms`
  - `end_ms`
  - `duration_ms`
  - `node_type`
  - `node_flags`
  - `summary`
  - short transcript excerpt
  - word count
  - aggregated speech-emotion evidence
  - aggregated audio-event evidence
  - inbound / outbound subgraph-local edges with edge type and target node ID
- recommended subgraph-level payload:
  - `subgraph_id`
  - `seed_node_id`
  - `source_prompt_ids`
  - ordered node list
  - local timeline span
- keep the payload graph-local:
  - do not include arbitrary distant graph neighbors outside the finalized local subgraph
  - do not include raw Phase 1 visual ledgers or renderability details in Phase 4 prompts
- if transcript excerpts must be trimmed for token budget, trim them deterministically:
  - preserve node summaries first
  - then preserve the seed node's transcript excerpt
  - then preserve transcript excerpts for boundary nodes
  - drop interior transcript detail before dropping structural metadata

Gemini output contract for subgraph review:

- Gemini must not invent node IDs, timestamps, edge types, or evidence not present in the provided subgraph payload
- Gemini must choose candidates only from the provided ordered node list
- Gemini may omit weak nodes from the middle of a candidate only when the remaining node set still forms a chronologically valid and semantically self-contained clip
- if Gemini proposes non-contiguous node selections, the implementation must either:
  - reject the candidate, or
  - deterministically fill the interior chronological gap only when every skipped node is purely connective and does not break semantic completeness
- recommended MVP rule:
  - require each candidate's `node_ids` to form one contiguous chronological span within the reviewed subgraph
- Gemini should return explicit `reject_all` when the subgraph does not contain a strong clip
- Gemini should also return a short failure reason for `reject_all`, such as:
  - weak payoff
  - incomplete setup
  - duplicate of a stronger nearby moment
  - overly context-dependent
  - low audience / trend value

Recommended structured output shape for subgraph review:

- Phase 4 should request strict structured JSON from Gemini for each reviewed local subgraph
- recommended response shape:

```json
{
  "subgraph_id": "sg_014",
  "seed_node_id": "node_007",
  "reject_all": false,
  "reject_reason": "",
  "candidates": [
    {
      "node_ids": ["node_007", "node_008", "node_009"],
      "start_ms": 12400,
      "end_ms": 43100,
      "score": 7.9,
      "rationale": "Clean setup-to-payoff sequence with strong reaction lift."
    }
  ]
}
```

- recommended validation rules for subgraph-review responses:
  - `subgraph_id` must match the request payload
  - `seed_node_id` must match the request payload
  - `reject_all=true` requires `candidates=[]`
  - `reject_all=false` requires at least one candidate
  - every `node_id` must belong to the provided ordered subgraph node list
  - `start_ms` and `end_ms` must match the chosen node span exactly
- if Gemini returns invalid structured output:
  - attempt one repair / reformat pass
  - if repair fails, treat the subgraph as `reject_all` and record the failure in debug output

Recommended candidate scoring dimensions during subgraph review:

- even if Gemini returns one blended score for MVP, the prompt should ask it to reason over distinct scoring dimensions
- recommended dimensions:
  - `hook_strength`
  - `payoff_strength`
  - `semantic_completeness`
  - `grammatical_completeness`
  - `self_containedness`
  - `interaction_dynamics`
  - `emotion_or_audio_lift`
  - `audience_or_trend_value`
- implementation note:
  - V3.1 may store only the final blended subgraph-review score in the canonical artifact
  - but prompts and optional debug artifacts should preserve these dimensions so ranking behavior can be tuned later

Subgraph deduplication before pooling:

- before sending any subgraph to Gemini, check for overlap with other subgraphs from the same run
- if two subgraphs share more than `70%` of their node IDs, drop the lower-scoring one
- score a subgraph by the sum of expansion scores of its included nodes
- this prevents Gemini from seeing the same strong moment twice under different prompts

Candidate deduplication after subgraph review:

- after Gemini returns candidate clips from all reviewed subgraphs, deduplicate candidate clips before the final pooled review pass
- treat two candidates as near-duplicates when either of the following is true:
  - their node-ID overlap is greater than `70%`
  - their wall-clock span IoU is greater than `0.70`
- keep the stronger candidate using this tie-break order:
  - higher subgraph-review score
  - stronger semantic completeness / payoff in the short rationale
  - shorter duration when both candidates cover essentially the same complete moment
- store duplicate relationships in debug output if available so retrieval quality can be inspected later

Seed-node diversity guardrail:

- after prompt retrieval and before subgraph construction, deduplicate seed nodes across prompt families
- if the same node is retrieved by multiple prompts, keep one canonical seed record and retain the full list of supporting prompt sources as provenance
- if one local chronological neighborhood dominates the top retrieval slots, enforce a light diversity cap so Phase 4 still explores at least `2-3` distinct regions when they exist
- recommended MVP rule:
  - no more than `2` seed nodes from the same `20s` timeline neighborhood unless the run has too few high-quality alternatives

Prompt and model-call budget guardrails:

- Phase 4 should enforce a per-run retrieval budget so candidate generation does not fan out uncontrollably on long videos
- recommended default budget:
  - at most `12` total retrieval prompts per run after all prompt-source expansion
  - at most `24` subgraphs sent to Gemini per run
  - at most `1` final pooled-review Gemini call per run
- if more eligible subgraphs exist than the run budget allows, keep only the highest-scoring non-duplicate subgraphs
- explicit user-query prompts may preempt low-priority meta-prompts when the run is at budget

Token-budget guardrails for Phase 4 model calls:

- each subgraph-review payload should stay under a deterministic token budget
- recommended MVP rule:
  - target no more than `6k-8k` prompt tokens per subgraph-review call
  - reserve enough output budget for up to `3` candidates plus rationale
- if a reviewed subgraph would exceed the token budget, shrink the payload in this order:
  - shorten transcript excerpts
  - shorten edge explanations to type + target only
  - drop low-value interior transcript detail
  - if still too large, trim the lowest-scoring expandable fringe nodes and recompute the final payload deterministically
- never trim away:
  - the seed node
  - either boundary node

Final pooled-candidate review:

- collect all candidates returned across all prompts and all local subgraphs
- send that pooled candidate set to Gemini in a final review pass
- Gemini should:
  - filter out irrelevant, weak, or redundant candidates
  - compare the remaining candidates against each other
  - rank the best candidates for standalone clip quality
- the ranked output of that final Gemini pass should be the contents of `clip_candidates.json`
- scores in the final ranked output should be interpreted as run-local ranking scores, not globally calibrated quality scores across all videos

Ranking policy for user-query vs organic retrieval:

- when a run includes an explicit user query, the final pooled review should treat query satisfaction as an additional ranking dimension
- query satisfaction should increase the rank of candidates that clearly answer the user's request
- however, query satisfaction should not force obviously weaker or incomplete clips above much stronger candidates unless the product surface is explicitly query-only
- recommended MVP behavior:
  - if a user query exists, allow the top ranked set to include both:
    - query-aligned winners
    - exceptional organic winners
  - but mark which candidates are query-aligned versus organically surfaced
- if there is no explicit user query, use the default organic ranking criteria only

Gemini output contract for pooled final review:

- Gemini must rank only candidates that survived deterministic validity and deduplication
- Gemini must not change candidate boundaries during the pooled final review pass
- Gemini may rerank, drop, or keep candidates, but it must preserve each surviving candidate's canonical `node_ids`, `start_ms`, and `end_ms`
- if Gemini believes two finalists should be merged or boundary-adjusted, that should be emitted only as an optional note for future tuning, not as a mutation of the canonical Phase 4 output

Recommended structured output shape for pooled final review:

- Phase 4 should also request strict structured JSON from Gemini for the pooled final review pass
- recommended response shape:

```json
{
  "ranked_candidates": [
    {
      "candidate_temp_id": "cand_tmp_004",
      "keep": true,
      "pool_rank": 1,
      "score": 8.4,
      "score_breakdown": {
        "overall_clip_quality": 8.7,
        "query_alignment": 0.0,
        "novelty_within_run": 7.9,
        "editorial_usability": 8.1,
        "confidence": 8.3
      },
      "rationale": "Strong hook-to-payoff arc with audience laughter at the peak."
    }
  ],
  "dropped_candidate_temp_ids": ["cand_tmp_008"]
}
```

- recommended validation rules for pooled-review responses:
  - every returned candidate ID must map to an input pooled candidate
  - `pool_rank` must be unique among kept candidates
  - kept candidates must be sorted into a valid contiguous rank order after deterministic tie-breaking
  - dropped candidates must not also appear as kept
- if Gemini returns invalid pooled-review output, fail the Phase 4 run and surface the validation error

Zero-candidate and low-candidate behavior:

- if every reviewed subgraph returns `reject_all`, Phase 4 should still emit a valid `clip_candidates.json`
- recommended zero-result behavior:
  - write an empty ranked array
  - write a small debug artifact explaining the dominant failure reasons across reviewed subgraphs
- if the run yields only `1-2` viable candidates, skip any forced diversity logic that would artificially demote the best available clip

Ranking should focus on:

- hook strength
- payoff strength
- semantic completeness
- grammatical completeness
- self-containedness
- interaction quality between speakers
- reaction dynamics
- disagreement / status shift / callback quality
- speech emotion / prosody intensity where relevant
- audio-event evidence such as laughter or applause
- audience resonance
- trend relevance

Recommended final pooled-review scoring dimensions:

- `overall_clip_quality`
- `query_alignment` when an explicit user query exists
- `novelty_within_run`
- `editorial_usability`
- `confidence`

Stable tie-breaking for final ranking:

- if two candidates receive effectively tied final scores, break ties deterministically in this order:
  - higher `semantic_completeness`
  - higher `self_containedness`
  - stronger `query_alignment` when a user query exists
  - shorter duration when both clips capture the same complete beat
  - earlier `start_ms`
  - lexicographic `clip_id`
- this tie-break order should be applied after model ranking so repeated runs do not reshuffle near-identical finalists

Primary output:

- `clip_candidates.json`

#### `clip_candidates.json` schema

Each entry in the ranked output array must include:

```json
{
  "clip_id": "clip_001",
  "start_ms": 12400,
  "end_ms": 43100,
  "node_ids": ["node_007", "node_008", "node_009"],
  "score": 8.4,
  "score_breakdown": {
    "overall_clip_quality": 8.7,
    "query_alignment": 0.0,
    "novelty_within_run": 7.9,
    "editorial_usability": 8.1,
    "confidence": 8.3
  },
  "rationale": "Strong hook-to-payoff arc with audience laughter at the peak.",
  "source_prompt": "meta_prompt_1",
  "seed_node_id": "node_007",
  "subgraph_id": "sg_014",
  "subgraph_node_ids": ["node_006", "node_007", "node_008", "node_009"],
  "pool_rank": 1,
  "source_prompt_ids": ["meta_prompt_1", "retention_cluster_2"],
  "query_aligned": false
}
```

- `clip_id`: unique identifier for the candidate, assigned during the final review pass
- `start_ms` / `end_ms`: wall-clock boundaries derived from the node span
- `node_ids`: ordered list of Phase 2 node IDs included in the clip
- `score`: Gemini's final quality score from the pooled review pass
- `score_breakdown`: optional but recommended run-local scoring dimensions used during final comparison
- `rationale`: short human-readable explanation of why this clip is strong
- `source_prompt`: which prompt type seeded this candidate (`meta_prompt_N`, `comment_cluster`, `retention_cluster`, `trend`, or `user_query`)
- `seed_node_id`: the node ID that was the original retrieval seed for this candidate's subgraph
- `subgraph_id`: identifier for the local subgraph from which this candidate emerged
- `subgraph_node_ids`: ordered list of all nodes provided to Gemini for that local subgraph
- `pool_rank`: the candidate's final rank position within the run-local pooled review output
- `source_prompt_ids`: all prompt sources that contributed to retrieval of the winning seed or equivalent local neighborhood
- `query_aligned`: whether the finalist was judged meaningfully aligned to an explicit user query in this run

Optional Phase 4 debug artifacts:

- `retrieval_prompts_debug.json`
  - records generated prompts, prompt type, and embedding/retrieval provenance
- `seed_nodes_debug.json`
  - records top retrieval matches per prompt before and after seed deduplication / diversity capping
- `local_subgraphs_debug.json`
  - records final subgraph membership and expansion scores
- `candidate_dedup_debug.json`
  - records near-duplicate candidate groups and which representative was kept
- `phase_4_summary.json`
  - records final counts:
    - prompts generated
    - seeds retained
    - subgraphs reviewed
    - candidates proposed
    - candidates removed as duplicates
    - finalists emitted

Phase 4 failure policy:

- if a subgraph-review call fails, the run may continue without that subgraph and record the failure in debug output
- if the pooled final-review call fails, times out, or returns invalid structured output, fail the Phase 4 run immediately
- do not silently repair pooled-final-review semantics with a deterministic ranking fallback in V3.1

Phase 4 evaluation hooks:

- although full evaluation belongs in the later evaluation section, Phase 4 should define the primary metrics it expects to emit for tuning
- recommended run-level metrics:
  - retrieval coverage:
    - fraction of reviewed subgraphs that yield at least one valid candidate
  - seed diversity:
    - number of distinct timeline regions represented in retained seeds
  - duplicate pressure:
    - fraction of proposed candidates removed by candidate-level deduplication
  - zero-result rate:
    - fraction of runs that emit no finalists
  - query-hit rate:
    - fraction of explicit-query runs whose final ranked set contains at least one `query_aligned` finalist
  - finalist compression:
    - ratio of finalists emitted to raw candidates proposed
- recommended annotation target for later evaluation:
  - whether a human editor judges each finalist to be:
    - usable as-is
    - usable with boundary trim
    - interesting but incomplete
    - not worth surfacing

3. `time span -> tracklet` within a shot

#### Voiceprints For Recurring Speakers

V3.1 should support an optional creator-scoped voiceprint registry built on pyannote's voiceprint and identify APIs.

Recommended use case:

- during onboarding, let the creator upload or record clean samples for themselves and optionally common recurring speakers
- a creator repeatedly uses Clypt
- during speaker-timeline editing, they rename one or more recurring speakers
- once a clean single-speaker region is confirmed, Clypt can offer to create and store a voiceprint for that named speaker
- if that person appears in a future recording, Clypt can use pyannote identification as a labeling accelerator

Recommended lifecycle:

1. onboarding may create an initial recurring-speaker voiceprint registry
2. later editing sessions may add more recurring speakers from confirmed clean samples
3. future runs may call `POST /v1/identify` with the stored voiceprints
4. identified speakers should still flow through normal review and correction in the grounding UI

Recommended pyannote-aligned constraints:

- one voiceprint per recurring speaker
- source audio must be a single-speaker region with no overlap
- sample length should be at most 30 seconds
- choose a region that reflects the person's normal speaking voice

Recommended product policy:

- this should be opt-in, not automatic
- store voiceprints securely in Clypt-owned storage because pyannote outputs are retained for only 24 hours
- treat identified labels as suggestions or bootstrap labels, not irreversible truth
- allow the user to approve, rename, or reject identified matches
- be conservative for guests unless the creator explicitly opts in and has the right to store those voiceprints

Good V3.1 usage:

- recurring hosts
- recurring co-hosts
- recurring collaborators who appear across uploads

Less suitable as an automatic default:

- one-off guests
- clips where the only clean sample is very short or overlapped
- any situation where consent or storage policy is unclear

#### Important Separation

Speaker participation must not be overloaded with camera direction.

Example:

- `speaker participation`: Speaker 2 is talking
- `camera intent`: follow or react on Speaker 1

That must be representable without contradiction.

Primary outputs:

- updated `participation_timeline.json`
- `camera_intent_timeline.json`

Speech emotion / prosody and audio event detection answer different questions:

Gemini can still interpret these signals semantically in Phase 2A, but should not be the primary detector for them.

### Phase 5 Detailed Participation Grounding Interaction And Data-Write Flow

This section defines the MVP interaction model for the three grounding modes:

1. `speaker_id -> tracklet` within a shot
2. `word span -> tracklet` within a shot
3. `time span -> tracklet` within a shot

The core design rule is:

- these are three UI entry points into one normalized write model
- they must not become three separate backend truth systems

#### Design Principles

- one canonical source of truth for user grounding writes
- materialized timelines are derived from user writes plus canonical media timelines
- explicit user writes always beat automated or default values
- speaker participation writes remain separate from camera intent writes
- speaker naming is orthogonal metadata, not a binding mechanism
- shot scope is the default scope for tracklet binding in MVP
- the UI should support fast shot-by-shot grounding without forcing the user to watch an entire shot before moving on
- safe visual defaults may exist for camera follow suggestions, but they must not be treated as automatic speaker-participation truth

#### UX Surfaces

The grounding UI should expose the same underlying entities in all modes:

- speaker chips:
  - `Speaker_00`, `Speaker_01`, etc.
  - optional human-readable names like `Host`, `Guest`
- shot strip / shot selector
- shot markers on the timeline so the user can jump directly to the next shot boundary during participation labeling
- visible bounding boxes / tracklet targets within the current shot
- diarized transcript with per-word highlighting
- time-range scrubber

Recommended Phase 2 UX rule:

- when the user is doing speaker participation labeling, the interface should show clear shot markers / shot boundaries on the timeline
- this lets the user label one shot, then jump directly to the next shot without sitting through the rest of the current shot
- this is especially important for shots with multiple visible people where the user already knows who is speaking early in the shot

Recommended safe prepopulation rule:

- if a shot has only one clear bounding box / tracklet, the UI may surface that as a likely visual follow target for later camera direction
- this should not be auto-written into the participation timeline as speaker truth
- the user must still explicitly confirm any speaker-participation mapping

#### Core Write Model

V3.1 should store raw user actions in an append-only grounding log, then build the derived participation timeline from that log.

Recommended write artifacts:

- `speaker_registry.json`
- `grounding_events.jsonl`
- `participation_timeline.json`

#### `speaker_registry.json`

Video-scoped speaker metadata:

```json
{
  "video_id": "abc123",
  "speakers": [
    {
      "speaker_id": "Speaker_00",
      "display_name": "Host"
    },
    {
      "speaker_id": "Speaker_01",
      "display_name": "Guest"
    }
  ]
}
```

#### `grounding_events.jsonl`

`Append-only` means:

- when the user confirms a new edit, the system adds a new record
- it does not destructively overwrite prior records
- the current truth is rebuilt from the ordered history of confirmed edits

This is useful because:

- users can start with broad mappings and then refine smaller sections
- corrections do not require fragile in-place mutation
- future training analysis can inspect how labels were created and revised

The log stores user actions like:

```json
{
  "event_id": "ge_00001",
  "created_at": "2026-04-01T20:11:00Z",
  "video_id": "abc123",
  "clip_candidate_id": null,
  "event_type": "bind_speaker_to_tracklet",
  "scope": {
    "shot_id": "shot_012"
  },
  "payload": {
    "speaker_id": "Speaker_00",
    "tracklet_id": "shot_012_tr_03"
  }
}
```

Derived participation truth should not be edited directly.
It should be materialized from these events.

#### Normalized Participation Assignment Shape

Regardless of how the user interacts, the system should normalize writes into assignment segments shaped like:

```json
{
  "start_ms": 1200,
  "end_ms": 4200,
  "speaker_ids": ["Speaker_00"],
  "tracklet_ids": ["shot_012_tr_03"],
  "binding_source": "speaker_id_map|word_span|time_span",
  "shot_id": "shot_012",
  "confidence": 1.0,
  "user_confirmed": true
}
```

Those normalized segments are then merged into `participation_timeline.json`.

#### Grounding Mode 1: `speaker_id -> tracklet`

##### User Flow

1. user enters a shot
2. user clicks a visible bounding box / tracklet
3. if the original shot is actually focused on a silent reaction, the user should still be able to bind the correct speaking participant rather than inheriting the visible reaction target
4. user clicks a speaker chip such as `Speaker_00`
5. optional: user renames that speaker to `Host`
6. user confirms the shot-local mapping

##### Intended Semantics

This does not mean:

- "follow this person with the camera"
- "this person owns every exact word in the shot with word-perfect precision"

It means:

- "when this diarized speaker is the active speaker in this shot, this is the correct tracklet"

This is intentionally a coarse default mapping for normal non-overlap cases.

It is designed for common easy cases such as:

- a podcast shot where one diarized speaker clearly corresponds to one visible person
- an interview shot where the same person remains the obvious speaking subject throughout the shot

It is not intended to be the highest-precision truth for:

- overlapping speech
- fast interruptions
- reaction-only moments
- exact word ownership

Those cases should be refined with:

- `word span -> tracklet` for precise word ownership
- `time span -> tracklet` for interval-level participation corrections

##### Write Behavior

The system should append:

- one `bind_speaker_to_tracklet` event
- optional `rename_speaker` event

Suggested event shape:

```json
{
  "event_type": "bind_speaker_to_tracklet",
  "scope": {
    "shot_id": "shot_012"
  },
  "payload": {
    "speaker_id": "Speaker_00",
    "tracklet_id": "shot_012_tr_03"
  }
}
```

The materializer should then:

- look up diarized speaker turns for `Speaker_00` inside `shot_012`
- create normalized participation segments only for the overlapping portions of those turns

This keeps the write semantically tight and avoids incorrectly labeling the whole shot if the speaker is silent for part of it.

#### Grounding Mode 2: `word span -> tracklet`

##### User Flow

1. user selects a shot
2. transcript view highlights words in sync with playback
3. user clicks or drags across a word span
4. user clicks a visible tracklet in that shot
5. user confirms the assignment

##### Intended Semantics

This is the highest-precision participation write.

It means:

- "these exact spoken words belong to this tracklet"

##### Write Behavior

The system should append a `bind_word_span_to_tracklet` event:

```json
{
  "event_type": "bind_word_span_to_tracklet",
  "scope": {
    "shot_id": "shot_012",
    "word_start_id": "w_000120",
    "word_end_id": "w_000156"
  },
  "payload": {
    "speaker_id": "Speaker_00",
    "tracklet_id": "shot_012_tr_03"
  }
}
```

The materializer should:

- resolve the selected words to `start_ms/end_ms`
- create an explicit normalized participation segment
- mark the segment as `binding_source=word_span`

This mode should take precedence over all broader mappings because it is the most explicit and precise.

#### Grounding Mode 3: `time span -> tracklet`

##### User Flow

1. user selects a shot or clip candidate section
2. user drags across a time range on the scrubber
3. user clicks a visible tracklet
4. optional: user also associates one or more active `speaker_id`s if the UI knows them
5. user confirms

##### Intended Semantics

This means:

- "during this exact time period, this tracklet is the correct participant mapping"

This is useful when:

- the transcript is inconvenient to use
- the user is reasoning visually
- the user wants to capture a short reaction or ambiguous portion by time rather than text

Important:

- `time span -> tracklet` is an interval-level participation label
- it should not automatically be treated as exact word ownership unless the selected time span corresponds to a precise word-backed write
- if multiple people are assigned within the same time span, the system should treat that as multi-participant activity in the interval, not as proof of exact per-word speaker ownership

##### Write Behavior

The system should append a `bind_time_span_to_tracklet` event:

```json
{
  "event_type": "bind_time_span_to_tracklet",
  "scope": {
    "shot_id": "shot_012",
    "start_ms": 2500,
    "end_ms": 3500
  },
  "payload": {
    "speaker_ids": ["Speaker_00"],
    "tracklet_id": "shot_012_tr_03"
  }
}
```

The materializer should create an explicit normalized participation segment with `binding_source=time_span`.

#### Speaker Renaming

Speaker renaming should be its own event type:

```json
{
  "event_type": "rename_speaker",
  "payload": {
    "speaker_id": "Speaker_00",
    "display_name": "Host"
  }
}
```

Rules:

- renaming is video-scoped for MVP
- renaming changes labels shown in the UI
- renaming does not itself create any participation mapping

#### Conflict Resolution Rules

The materializer for `participation_timeline.json` should apply writes in precedence order.

##### Primary Rule: More Specific Scope Beats Less Specific Scope

The core precedence rule should be based on specificity, not on a hardcoded preference for one UI mode over another.

Highest to lowest specificity:

1. exact `word span -> tracklet`
2. bounded `time span -> tracklet`
3. shot-scoped `speaker_id -> tracklet`
4. no user binding

Rationale:

- word span is the narrowest and most transcript-anchored instruction
- time span is explicit but broader than a precise word selection
- shot speaker mapping is intentionally coarse and optimized for fast setup

This lets the user:

- start with broad shot-level mappings
- then refine smaller ranges without deleting the broader work

##### Precedence Within Same Write Type

- latest confirmed user write wins

This should let the user correct mistakes without destructive mutation of the event log.

##### Segment Splitting Behavior

If a narrower higher-precedence write overlaps a broader lower-precedence write, the materializer should split the broader segment around it rather than dropping it wholesale.

Example:

- shot-level mapping says `Speaker_00 -> tr_03` for the shot
- later word-span mapping corrects a short phrase to `tr_05`

The final participation timeline should become:

- before phrase: `tr_03`
- phrase: `tr_05`
- after phrase: `tr_03`

#### Practical Interpretation

This precedence should be understood as:

- broad writes establish defaults
- narrower writes act as corrections or refinements
- later writes only win automatically when they operate at the same specificity level

#### Unbinding / Correction

The user must be able to remove or override prior writes.

Recommended event:

```json
{
  "event_type": "clear_binding_range",
  "scope": {
    "shot_id": "shot_012",
    "start_ms": 2500,
    "end_ms": 3500
  }
}
```

This avoids destructive rewrites and keeps the event log useful for future training analysis.

#### Materialized Participation Timeline

The final `participation_timeline.json` should be a merged non-overlapping timeline of user-confirmed speaker participation segments.

Suggested shape:

```json

    "speaker_ids": ["Speaker_00"],
    "speaker_names": ["Host"],

    "shot_id": "shot_012",
    "user_confirmed": true
  }

```

#### Relationship To Camera Intent

Participation writes and camera intent writes must remain separate.

Participation says:

- who is speaking / participating

Camera intent says:

- who to emphasize visually

So later, after candidate selection, the user may create camera intent segments such as:

```json
{
  "start_ms": 2100,
  "end_ms": 2500,
  "intent": "reaction",
  "primary_tracklet_id": "shot_012_tr_07"
}
```

even if the participation timeline for that same segment says:

- `Speaker_00 -> shot_012_tr_03`

That is valid and expected.

#### Candidate-Scoped Writes

Because participation grounding happens after candidate discovery in V3.1, grounding events should optionally include:

- `clip_candidate_id`

This allows the system to distinguish:

- future video-global grounding if the product ever adds it
- current finalist-local grounding

Recommended rule:

- candidate-local writes only materialize participation segments inside the candidate interval unless the user explicitly expands scope

#### Implementation Notes

- all three grounding modes should write into one event stream
- the materializer should be deterministic and replayable
- derived timelines should be cheap to rebuild after every edit
- preview state can exist in the UI, but only confirmed edits should hit the event log
- camera intent should use the same event-sourcing pattern in a separate `camera_intent_events.jsonl` file

### Phase 5 Detailed Camera Intent Interaction And Data-Write Flow

This section defines the MVP interaction model for post-candidate camera direction:

- `follow`
- `reaction`
- `split`
- `wide`

The design goal is:

- let users direct the visual emphasis of finalist clips without mutating speaker participation truth

In this section:

- `camera intent` means the user's structured visual direction chosen through the UI
- it does not mean a chat command
- it is created through click / drag interactions on finalist clips

#### Design Principles

- camera intent is a separate write stream from participation grounding
- camera intent is candidate-scoped by default
- camera intent writes are time-bounded, never assumed global unless the user explicitly expands scope
- shot changes should be respected by the compiler even if the user's intent segment crosses multiple shots
- the UI should feel simple, but the write model should be expressive enough to support rerendering without re-grounding or re-ranking

#### Core Write Model

Recommended write artifacts:

- `camera_intent_events.jsonl`
- `camera_intent_timeline.json`
- `render_plan.json`

#### `camera_intent_events.jsonl`

Camera intent should use the same append-only event log strategy as participation.

Example:

```json

  "created_at": "2026-04-01T21:00:00Z",
  "video_id": "abc123",
```

#### Materialized Camera Intent Segment

All camera intent writes should normalize to:

```json
{
  "start_ms": 1200,
  "end_ms": 4200,

  "primary_tracklet_id": "shot_012_tr_03",
  "secondary_tracklet_id": null,
  "clip_candidate_id": "cand_004",
  "user_confirmed": true
}
```

`primary_tracklet_id` and `secondary_tracklet_id` should be nullable depending on the intent type.

#### Intent Types

##### 1. `follow`

Meaning:

- keep the camera focused on this tracklet

Use cases:

- the speaking person should be the focus
- a single dominant on-screen subject should anchor the shot

Required fields:

- `primary_tracklet_id`

##### 2. `reaction`

Meaning:

- the semantically important visual target is this tracklet's reaction, even if someone else is speaking

Use cases:

- podcast guest says something, but host reaction is the payoff
- disagreement moment where listener face sells the clip

Required fields:

- `primary_tracklet_id`

Important:

- `reaction` is not just a label alias for `follow`
- it is a semantic instruction that the renderer/compiler may use for UI copy, analytics, and future layout policy
- it is the main way for the user to intentionally show a non-speaking person's face during someone else's words

##### 3. `split`

Meaning:

- present two tracklets together in an explicit split layout

Use cases:

- both participants matter
- disagreement / banter moments
- simultaneous reaction and speech both need to remain visible

Required fields:

- `primary_tracklet_id`
- `secondary_tracklet_id`

MVP rule:

- the order should be preserved in the write model so the UI can keep a stable top/bottom or left/right semantic ordering if needed

##### 4. `wide`

Meaning:

- do not aggressively follow a single participant
- preserve wider framing

Use cases:

- both people are visible and spatial context matters
- shot is visually stable and crop-follow would reduce quality
- the user wants to preserve a shared composition

Required fields:

- no tracklet IDs required

#### User Interaction Flows

The MVP camera-direction interaction model should feel like:

1. select when
2. select who
3. select how

Where:

- `when` = choose the interval by dragging words in the transcript or dragging a time range on the timeline
- `who` = click one or two visible bounding boxes / tracklets in the preview
- `how` = choose `Follow`, `Reaction`, `Split`, or `Wide`

This matters because the user may want to show someone who is not currently speaking.

Example:

- a guest speaks for a few words
- the host's reaction is more important than the speaker's face
- the user drags over those words in the transcript
- clicks the host's bounding box
- chooses `Reaction`

Result:

- participation timeline still says the guest is speaking
- camera intent says to emphasize the host for that short interval

#### Conflict Resolution Rules

##### Primary Rule: More Specific Scope Beats Less Specific Scope

Use the same principle as participation:

- narrower time ranges override broader defaults
- later writes win within the same specificity level

##### Intent Replacement

Camera intent is singular per instant in MVP.

That means:

- `follow`, `reaction`, `split`, and `wide` are mutually exclusive in the materialized timeline

If a user writes a narrower `reaction` inside a broader `follow`, the compiler should split the broader segment around it.

Example:

- `0-10s follow(tr_03)`
- `3-5s reaction(tr_07)`

Materialized result:

- `0-3s follow(tr_03)`
- `3-5s reaction(tr_07)`
- `5-10s follow(tr_03)`

##### Clear / Remove Intent

Recommended event:

```json
{

  "scope": {
    "start_ms": 3000,
    "end_ms": 5000
  }
}
```

#### Candidate Scope And Expansion

For MVP, camera intent should default to `clip_candidate_id` scope.

Meaning:

- the user's direction is assumed to apply only to the current finalist clip
- it should not silently become a video-global preference

Optional later UX:

- "apply to entire source span"
- "reuse this intent on similar overlapping finalists"

But that is out of scope for the initial MVP write model.

#### Default Camera Behavior From Participation

Participation provides the automatic camera baseline.
Custom camera choices override that baseline only where the user explicitly sets them.

Core rule:

- if custom camera intent exists for an interval, use it
- otherwise, derive the camera choice from the participation timeline

##### MVP Default Policy Table

| Participation State | Default Camera Behavior | Why |
|---|---|---|
| One grounded active participant | `follow(primary tracklet)` | simplest and usually correct |
| One grounded active participant, but no valid visible target at part of the interval | `wide` for unresolved portion | avoid hallucinating a target |
| No grounded participant, but the source shot has one obvious visual subject | `follow(obvious visible tracklet)` as a soft visual default | useful as a camera suggestion only, not as speaker truth |
| Two grounded active participants, both visible, geometry supports split | `split(tracklet_a, tracklet_b)` | both matter and can be shown clearly |
| Two grounded active participants, but split geometry is poor | `wide` | safer than bad split framing |
| More than two active participants | `wide` | MVP should avoid overcomplicated automatic layouts |
| No grounded participant | `wide` | no reliable follow target |
| One active speaker plus another interval-level active participant from overlap or coarse labeling | `split` if composition is clean, else `wide` | both matter, but avoid fake precision |
| One active speaker, one non-speaking but important reactor | `follow(active speaker)` by default | reaction importance is usually a human editorial choice unless explicitly overridden |

##### Interpretation

Participation answers:

- who is active?

Default camera policy answers:

- if the user gives no extra direction, what is the safest and most reasonable shot choice?

Custom camera intent answers:

- what does the user actually want the viewer to look at?

So:

- participation drives automation
- camera intent drives editorial control

##### Examples

###### Example 1: Simple speaking case

- participation:
  - `12.0s -> 16.0s Speaker_00 -> tr_03`
- no custom camera
- output:
  - `follow(tr_03)`

###### Example 2: Reaction override

- participation:
  - `12.0s -> 16.0s Speaker_00 -> tr_03`
- default:
  - `follow(tr_03)`
- user selects `13.1s -> 14.0s` and sets `reaction(tr_07)`

Final camera:

- `12.0s -> 13.1s follow(tr_03)`
- `13.1s -> 14.0s reaction(tr_07)`
- `14.0s -> 16.0s follow(tr_03)`

###### Example 3: Two-speaker overlap

- participation:
  - `20.0s -> 22.0s Speaker_00 -> tr_03`
  - `20.0s -> 22.0s Speaker_01 -> tr_06`
- default:
  - `split(tr_03, tr_06)` if visually clean
  - otherwise `wide`

Possible user overrides:

- `follow(tr_03)`
- `reaction(tr_06)`
- `wide`

###### Example 4: Ambiguous coarse interval

- participation:
  - `30.0s -> 31.5s tr_03 and tr_06 both active`
- default:
  - `split` or `wide`

If the user knows one face matters more, they can override with:

- `reaction(tr_06)`
- `follow(tr_03)`

#### Shot Boundary Handling

##### User Experience Rule

Users should be allowed to create one camera-choice segment that crosses shot boundaries.

They should not be forced to think in shot-sized chunks while editing.

##### Compiler Rule

The compiler must split camera-choice segments at shot boundaries before producing the final render instructions.

Reason:

- tracklets are shot-local in V3.1
- a `follow` or `reaction` target cannot be assumed valid across a cut
- `split` layouts may need to rebind or degrade after a cut

Example:

- user writes `2000-7000ms follow(shot_012_tr_03)`
- shot changes at `5400ms`

Compiler output should become:

- `2000-5400ms follow(shot_012_tr_03)`
- `5400-7000ms unresolved_follow_after_cut`

Then the compiler should resolve the second segment using the rules below.

#### Compiler Rules: Camera Intent -> Render Plan

The render compiler should read:

- `camera_intent_timeline.json`
- `participation_timeline.json`
- `canonical_timeline.json`
- `shot_tracklet_index.json`
- `tracklet_geometry.json`

and produce shot-by-shot render instructions.

`Shot-by-shot render instructions` means:

- the user's higher-level camera choices have been broken into segments that align with real shot boundaries
- each segment has been checked against the shot-local tracklets that actually exist there
- invalid targets after a cut have been degraded or marked for review
- the renderer receives concrete instructions it can execute directly

##### Follow Resolution

For a `follow(tracklet)` segment:

- if the target tracklet exists in that shot segment, compile direct follow instructions
- if the segment crosses a cut and no valid corresponding target exists after the cut, do not invent a fake cross-shot identity
- degrade to:
  - `wide`, or
  - require user correction if this is a finalist preview step

MVP recommendation:

- preview degrades to `wide`
- UI should visually mark the segment as needing review

##### Reaction Resolution

For a `reaction(tracklet)` segment:

- behave like `follow` geometrically
- preserve the semantic tag `reaction` in the render plan

Reason:

- later analytics / UX should know this was an intentional reaction-directed shot, not just a normal follow

##### Split Resolution

For a `split(tracklet_a, tracklet_b)` segment:

- if both targets are valid and visible in the shot, compile split layout
- if one target becomes unavailable after a cut:
  - degrade to `follow` on the surviving valid target only if the user explicitly selected a fallback preference in future versions
  - otherwise degrade to `wide` and mark for review

MVP recommendation:

- degrade to `wide` and mark for review

##### Wide Resolution

For a `wide` segment:

- preserve wide framing
- no tracklet validation required
- shot cuts do not invalidate the intent

- re-running graph construction
- re-running clip ranking

This is one of the key product advantages of the V3.1 architecture.

## Materializers And Compilers

V3.1 should introduce explicit derived-state builders.

### Materializers

Materializers take append-only user event logs and build current truth artifacts.

Required materializers:

- `materialize_participation.py`
- `materialize_camera_intent.py`

These should be:

- deterministic
- replayable
- cheap to rerun after every confirmed edit

### Compilers

Compilers turn user-facing abstractions into renderer-facing instructions.

Required compiler:

- `compile_camera_plan.py`

Inputs:

- participation timeline
- camera intent timeline
- shot boundaries
- tracklet geometry
- candidate clip boundaries

Output:

- `render_plan.json`

Important distinction:

- materializers produce current truth
- compilers produce executable instructions

## Service Boundaries

Recommended rule of thumb:

- `timeline/` owns canonical aligned evidence
- `grounding/` owns participation truth from user labeling
- `semantics/` owns merged-node construction and node classification
- `graph/` owns graph topology and storage
- `candidates/` owns retrieval, local expansion, clip assembly, and ranking
- `camera/` owns default camera policy plus user visual direction
- `render/` owns execution of final rendering
- `api/` owns transport and lifecycle only

This separation is important so that:

- rerender does not touch semantics
- participation relabeling does not require rerunning node construction or classification
- camera edits do not mutate graph truth

## Migration Sequence

Recommended implementation sequence:

1. create `backend/pipeline/v3_1/contracts.py` and `backend/pipeline/v3_1/artifacts.py`
2. build `timeline/` and produce run-scoped V3.1 artifacts from existing Phase 1 outputs
3. build `semantics/` merged-node construction, node classification, reconciliation, and node embeddings

5. build `candidates/` retrieval, local subgraph selection, and Gemini-based candidate ranking
6. build `grounding/` event logs and materializers for finalist participation

8. adapt renderer through `render/` and `render_plan.json`
9. add API endpoints for grounding, camera choices, render-plan compilation, and rerender
10. wire `run_pipeline.py` or a separate runner into `backend/pipeline/v3_1/orchestrator.py`
