# Clypt V3.1 Refactor Spec

**Status:** Draft v1  
**Date:** 2026-04-01  
**POC:** Rithvik  

## TL;DR

This refactor should make Clypt V3.1 an adaptive two-lane system:

- lane 1: semantic understanding, graph construction, and clip candidate ranking
- lane 2: human-guided speaker participation grounding and post-candidate camera direction

Core decisions locked in for this spec:

- remove LR-ASD entirely
- keep diarized speaker IDs as the global audio identity backbone
- keep shot-local tracklets as the visual identity backbone
- use overlapping windows only as compute scaffolding, never as final nodes
- create final graph nodes as merged, refined semantic spans
- separate `speaker participation` from `camera intent`
- use full early identity grounding only for shorter or more dynamic videos per routing policy
- for MVP, use a dedicated speech emotion/prosody model and a dedicated audio event model instead of relying on Gemini to infer all tone/audio signals

## Goals

- improve semantic and grammatical clip boundaries
- reduce cost and latency compared with broad multimodal reasoning on every segment
- preserve Clypt's graph-first value:
  - semantic relationships across moments
  - audience resonance reasoning
  - trend relevance reasoning
- preserve the product ethos of "breaking down the video for the user"
- support explicit user-directed rerendering for follow, reaction, split, and wide layouts

## Non-Goals

- preserving compatibility with the current Phase 2A-5 implementation
- preserving LR-ASD or any audiovisual active-speaker model path
- building the future supervised training loop in MVP
- building fully open-ended keyframed camera direction in MVP

## Hard Constraints

- no LR-ASD at all
- no render-time fallback that tries to infer true speaker binding from audiovisual synchrony
- fixed windows are compute shards only
- speaker participation and camera direction must remain separate artifacts
- camera intent is always post-candidate-selection
- for long/static videos, speaker participation grounding may remain sparse or empty until finalists

## Routing Policy

V3.1 should route videos into one of four modes:

| Content Type | Duration | Grounding Mode |
|---|---:|---|
| Dynamic | `< 30 min` | full identity grounding before candidate selection |
| Dynamic | `>= 30 min` | finalist identity grounding only |
| Static / podcast | `< 15 min` | full identity grounding before candidate selection |
| Static / podcast | `>= 15 min` | finalist identity grounding only |

Routing sources:

- onboarding / creator category
- creator priors from known channel/content style
- cheap video metadata heuristics
- shot-change rate / motion heuristics from extracted media
- optional comment/context heuristics if available upstream

## Core Model

### 1. Canonical Transcript Timeline

This is the primary alignment backbone for downstream semantics.

It should contain:

- pyannote word-level transcription with timestamps
- diarized speaker turns
- sentence boundaries

For V3.1, the canonical transcript backbone should come from the full pyannote stack via API, including Precision-2 diarization and pyannote STT orchestration outputs where available.
That means the primary transcript and diarization source should be pyannote rather than a split transcription-plus-diarization design.

Planned pyannote usage:

- default path: `POST /v1/diarize` with `model=precision-2`
- set `transcription: true` to request speaker-attributed transcription outputs
- identification path: `POST /v1/identify` when stored recurring voiceprints are available for likely speakers in the run
- overlapping diarization as the default source of truth for speaker-turn structure
- optional `exclusive: true` only when a simplified non-overlap view is explicitly needed for reconciliation or UX

Routing rule:

- if no recurring voiceprints are available for the run, use `diarize`
- if recurring voiceprints are available, use `identify`

Important:

- voiceprints support speaker identification only
- they should be treated as a labeling accelerator and bootstrap signal
- they should not replace user review or the participation-grounding workflow

Relevant pyannote outputs for V3.1:

- `diarization`
- `wordLevelTranscription`
- `turnLevelTranscription`

`turnLevelTranscription` should be treated as a readable convenience layer.
It does not replace sentence boundaries because a single speaker turn can still contain multiple sentences, setup/payoff structures, or topic pivots.

Planned sentence-boundary strategy:

- derive sentence, question, and exclamation boundaries from pyannote's punctuated `turnLevelTranscription`
- map those sentence boundaries back onto the word-level timeline

Explicit non-goal for V3.1:

- do not compute a separate clause-boundary artifact in the initial version

### 2. Visual Shot / Tracklet Index

This is the primary visual structure layer.

It should contain:

- shot boundaries
- shot-local tracklets
- lightweight tracklet descriptors (human-friendly summary)
- tracklet geometry per frame (renderable motion data)

For MVP, `lightweight tracklet descriptors` should mean cheap geometry and UI metadata only.
They should not assume a face-detection pipeline.

Recommended MVP descriptor fields:

- `tracklet_id`
- `shot_id`
- `start_ms`
- `end_ms`
- `visible_duration_ms`
- `visible_fraction`
- `median_bbox_center`
- `median_bbox_size`
- `scale_label` with values such as `close|medium|wide`
- `representative_thumbnail_uris`

Not part of the MVP descriptor set:

- face-presence
- face-visible fraction
- face-quality signals
- dominant_colors

Non-goal:

- global face identity computation is not required for speaker truth in V3.1 because diarization already provides the global audio identity backbone

### 3. Speech Emotion / Prosody Timeline

This should be generated by a dedicated speech emotion model such as `emotion2vec+`.

It captures how words are said:

- excited
- tense
- calm
- frustrated
- vulnerable
- intense / flat delivery
- prosodic emphasis changes

Planned V3.1 method:

- run emotion2vec on sentence-level segments
- derive those sentence segments from punctuated pyannote `turnLevelTranscription`
- align each sentence-level emotion reading back to the canonical word timeline and dominant speaker in that sentence span

Reason for this choice:

- more semantically meaningful than coarse turn-level emotion
- still much simpler than frame-level alignment
- better aligned with clip-boundary and semantic-node reasoning
- good enough for V3.1 while preserving the option to move to finer-grained windowed emotion analysis later

### 4. Audio Event Timeline

This should be generated by a dedicated audio event model such as `YAMNet`.

`YAMNet` is a general audio-event classifier, but V3.1 should use it as a curated reaction / crowd / music-event layer rather than as a generic 521-class dump.

It captures what sound event occurred:

- laughter
- chuckle
- applause
- cheering
- gasp
- sigh
- cough
- whispering-like events
- music swell
- notable music change events

Planned V3.1 curation policy using official YAMNet class names:

- strong keep classes for semantic prompting:
  - `Whispering`
  - `Laughter`
  - `Giggle`
  - `Snicker`
  - `Belly laugh`
  - `Chuckle, chortle`
  - `Crying, sobbing`
  - `Whimper`
  - `Wail, moan`
  - `Sigh`
  - `Groan`
  - `Grunt`
  - `Gasp`
  - `Shout`
  - `Bellow`
  - `Whoop`
  - `Yell`
  - `Screaming`
  - `Clapping`
  - `Cheering`
  - `Applause`
  - `Chatter`
  - `Crowd`
  - `Hubbub, speech noise, speech babble`
  - `Cough`
  - `Throat clearing`
  - `Sneeze`
  - `Sniff`
- music-evidence classes to keep for downstream change detection:
  - `Music`
  - `Background music`
  - `Theme music`
  - `Soundtrack music`
  - `Sad music`
  - `Tender music`
  - `Exciting music`
  - `Scary music`
- route-specific / consider-later classes:
  - `Singing`
  - `Choir`
  - `Chant`
  - `Rapping`
  - `Humming`
  - `Beatboxing`
  - `Pop music`
  - `Hip hop music`
  - `Rock music`
  - `Heavy metal`
  - `Punk rock`
  - `Progressive rock`

Ignore by default in V3.1:

- generic speech classes such as `Speech`, `Conversation`, and `Narration, monologue`
- most environmental / animal / machine / household / weather classes
- genre-only music classes unless route-specific logic later proves them useful

Normalization policy:

- do not persist raw YAMNet patch outputs as product artifacts
- convert patch-level detections into merged event spans in `audio_event_timeline.json`
- use class-specific thresholds rather than one global threshold
- require short contiguous support for noisy classes unless a single patch is very high confidence
- merge nearby same-class detections into one span
- if two or more different kept classes are simultaneously active and each passes its own threshold, keep all of them

Music policy:

- do not emit one giant flat `music present` span for the whole video
- use music-related classes primarily to derive editorially useful events such as:
  - `music_onset`
  - `music_offset`
  - `music_swell`
  - `music_drop`
  - `music_intensity_change`
- this keeps long monotone bed music from polluting the semantic layer while still preserving meaningful shifts

### 5. Semantic Span Graph

The graph should be built from final merged refined spans, not raw windows.

### 6. Speaker Participation Timeline

This answers:

> who (which tracklet(s)) is actually speaking or participating in this span?

This is only fully populated:

- before ranking for `full identity grounding` routes
- after ranking for finalists in `finalist identity grounding` routes

### 7. Camera Intent Timeline

This answers:

> who (which tracklet(s)) or what should the render emphasize in this span?

Supported MVP intents:

- `follow(tracklet)`
- `reaction(tracklet)`
- `split(tracklet_a, tracklet_b)`
- `wide`

This is always post-candidate-selection.

In this spec, `camera intent` means a structured choice made through the UI, not a chat instruction or natural-language prompt.

### 8. Render Plan

The render plan turns higher-level camera choices into concrete shot-by-shot render instructions.

It compiles:

- candidate boundaries
- shot geometry
- speaker participation
- camera intent

into shot-by-shot rendering instructions.

## Identity Grounding Model

### Global Truth

- global audio identity: diarized `speaker_id`
- local visual identity: `shot-local tracklet_id`

Optional identification enrichment:

- pyannote voiceprints for known recurring speakers

Important:

- voiceprints are for identification only
- they do not improve diarization accuracy directly
- they should enrich speaker labeling, not replace the participation-grounding model

### Grounding Methods

The user can ground speaker participation in three ways:

1. `speaker_id -> tracklet` within a shot
2. `word span -> tracklet` within a shot using the diarized transcript UI
3. `time span -> tracklet` within a shot

The user should also be able to rename `speaker_id`s with human-readable names.

### Voiceprints For Recurring Speakers

V3.1 should support an optional creator-scoped voiceprint registry built on pyannote's voiceprint and identify APIs.

Recommended use case:

- during onboarding, let the creator upload or record clean samples for themselves and optionally common recurring speakers
- a creator repeatedly uses Clypt
- during speaker-timeline editing, they rename one or more recurring speakers
- once a clean single-speaker region is confirmed, Clypt can offer to create and store a voiceprint for that named speaker
- if that person appears in a future recording, Clypt can use pyannote identification as a labeling accelerator

Recommended lifecycle:

1. onboarding may create an initial recurring-speaker voiceprint registry
2. later editing sessions may add more recurring speakers from confirmed clean samples
3. future runs may call `POST /v1/identify` with the stored voiceprints
4. identified speakers should still flow through normal review and correction in the grounding UI

Recommended pyannote-aligned constraints:

- one voiceprint per recurring speaker
- source audio must be a single-speaker region with no overlap
- sample length should be at most 30 seconds
- choose a region that reflects the person's normal speaking voice

Recommended product policy:

- this should be opt-in, not automatic
- store voiceprints securely in Clypt-owned storage because pyannote outputs are retained for only 24 hours
- treat identified labels as suggestions or bootstrap labels, not irreversible truth
- allow the user to approve, rename, or reject identified matches
- be conservative for guests unless the creator explicitly opts in and has the right to store those voiceprints

Good V3.1 usage:

- recurring hosts
- recurring co-hosts
- recurring collaborators who appear across uploads

Less suitable as an automatic default:

- one-off guests
- clips where the only clean sample is very short or overlapped
- any situation where consent or storage policy is unclear

### Important Separation

Speaker participation must not be overloaded with camera direction.

Example:

- `speaker participation`: Speaker 2 is talking
- `camera intent`: follow or react on Speaker 1

That must be representable without contradiction.

### Phase 2: Timeline Foundation And Routing

Build the full aligned foundation:

- canonical transcript timeline
- shot list
- shot-local tracklet index
- speech emotion / prosody timeline
- audio event timeline
- route decision:
  - dynamic vs static
  - full grounding vs finalist grounding

Primary outputs:

- `canonical_timeline.json`
- `speech_emotion_timeline.json`
- `audio_event_timeline.json`
- `shot_tracklet_index.json`
- `tracklet_geometry.json`
- `route_decision.json`

### Phase 2.5: Identity Grounding Gate

For `full identity grounding` routes only:

- pause before semantic ranking
- collect user participation labels
- populate participation timeline before candidate ranking

For `finalist identity grounding` routes:

- skip this phase initially
- defer participation grounding until finalists are chosen

Primary output:

- `participation_timeline.json`

### Phase 3: Proposal Generation, Merge, And Final Node Creation

Phase 3 should be broken into three sub-stages.

#### Phase 3A: Local Proposal Extraction

Create overlapping compute windows over the canonical timeline.

Example:

- `12s` windows with `6s` stride

These windows are internal compute scaffolding only.

For each window, collect:

- transcript words in range
- diarized speaker turns in range
- speech emotion / prosody events in range
- audio events in range
- optional visual summary context if routed to multimodal or escalated

Gemini generative calls belong primarily here.

Gemini's job in Phase 3A:

- emit local candidate semantic moments
- identify likely:
  - hook
  - claim
  - payoff
  - topic shift
  - reaction beat
  - tension/release beat
  - high-resonance interaction beat

Semantic pivots such as topic shifts should be discovered here during proposal extraction rather than being precomputed as a separate Phase 2 artifact.

Gemini should return local candidate spans referenced by either:

- local offsets within the window, or
- local word indices / word ids within the window slice

These are local proposals, not final nodes.

#### Phase 3B: Proposal Merge And Boundary Refinement

Phase 3B should be primarily deterministic.

Step 1: convert local proposals to global timestamps

Meaning:

- if a window spans `60000 -> 72000 ms`
- and Gemini proposes `4200 -> 9100 ms` relative to that window
- the global candidate span becomes `64200 -> 69100 ms`

If the proposal is word-index-based instead, map those words back to canonical `word_id`s and derive the real global times from the canonical timeline.

Step 2: compute temporary proposal-level embeddings

These are not the durable graph embeddings.
They are only for merge / dedupe.

Use them alongside deterministic gates:

- temporal overlap
- transcript-word overlap
- sentence overlap
- semantic similarity

Merge proposals when they are effectively the same underlying moment.

Step 3: refine merged boundaries

Refinement must:

- snap to real word boundaries
- avoid broken sentence endings
- preserve setup -> payoff when both are needed for semantic completeness
- preserve reaction beats when they are part of the same moment

Interrupted-sentence policy:

- the canonical timeline should remain literal and chronological
- if a speaker is interrupted and later resumes, the raw timeline should preserve those as separate chronological segments
- Phase 3 is allowed to merge across a short interruption when the resumed speech clearly completes the earlier thought and doing so improves semantic completeness

After refinement, the merged refined spans become the final semantic nodes.

#### Phase 3C: Final Node Embeddings

Once the final nodes exist, compute durable node embeddings for:

- graph construction
- retrieval
- audience signal linking
- trend linking
- later semantic search

These are distinct from the temporary proposal embeddings used in Phase 3B.

Primary output:

- `semantic_nodes.json`

Optional debug/checkpoint outputs:

- `proposal_debug.json`
- `merge_debug.json`

### Phase 4: Graph Construction

Build the graph from final semantic nodes.

Node types may include:

- semantic moment
- topic / entity
- audience signal
- trend signal
- speaker / character participation when grounded

Edge types may include:

- semantic similarity
- temporal adjacency
- setup -> payoff
- elaborates / supports
- contradiction / subversion
- callback / recurrence
- audience resonance links
- trend relevance links
- participation links when available

Important:

- speaker / character participation is graph enrichment only
- it is not required to be fully present on every node before ranking

Primary outputs:

- `semantic_graph_nodes.json`
- `semantic_graph_edges.json`

### Phase 5: Candidate Ranking

Rank final semantic spans or span groups.

Do not optimize for renderability risk in MVP.
The user will always provide participation grounding and camera intent for finalists at minimum.

Ranking should focus on:

- hook strength
- payoff strength
- semantic completeness
- grammatical completeness
- self-containedness
- interaction quality between speakers
- reaction dynamics
- disagreement / status shift / callback quality
- speech emotion / prosody intensity where relevant
- audio-event evidence such as laughter or applause
- audience resonance
- trend relevance

Primary output:

- `clip_candidates.json`

### Phase 5.5: Finalist Grounding

For `finalist identity grounding` routes:

- ask the user to ground participation only on shortlisted candidates

For all routes:

- ask the user to create camera intent only after candidate clips exist

Outputs:

- updated `participation_timeline.json`
- `camera_intent_timeline.json`

### Phase 6: Render Planning And Render

Compile:

- chosen candidate clip boundaries
- shot boundaries
- tracklet geometry
- participation timeline
- camera intent timeline

into:

- shot-by-shot follow instructions
- reaction-follow intervals
- split intervals
- wide intervals

Primary output:

- `render_plan.json`

## Embedding Strategy

### Principle

Embeddings should support understanding and merge quality, but they should not be the sole mechanism that creates nodes.

### Two Embedding Layers

#### 1. Temporary Proposal Embeddings

Used during Phase 3B for:

- dedupe
- merge
- near-duplicate suppression across overlapping windows

These are intermediate and do not need to be durable product artifacts.

#### 2. Durable Final Node Embeddings

Used after refinement for:

- graph storage
- retrieval
- trend / audience reasoning
- semantic search

### Modality Policy

Default policy:

- text + audio first

Escalation to multimodal should be selective, not broad.

Cheap escalation triggers only for MVP:

- high shot-change rate
- high motion / tracklet churn
- OCR density spike
- proposal disagreement across neighboring windows
- high-value but uncertain candidate span
- route says `dynamic`

Do not use expensive global visual-semantic disagreement checks as a baseline MVP requirement.

If later desired, those can be added only for shortlisted uncertain spans.

## Audio Understanding Policy

### Backbone Choice

V3.1 should use the full pyannote stack as the transcription and diarization backbone.

Planned usage:

- pyannote STT orchestration / transcription outputs for word-level and turn-level transcript artifacts
- pyannote diarization outputs for overlapping speaker turns and speaker IDs
- pyannote API as the primary source for the canonical transcript timeline

This replaces the earlier split assumption of:

- separate ASR backbone
- separate diarization backbone

### Why Two Separate Audio Sidecar Models

Speech emotion / prosody and audio event detection answer different questions:

- emotion / prosody: how speech is delivered
- audio events: what reaction / crowd / music-related sound occurred

These should remain separate artifacts in MVP.

### Recommended MVP Stack

- transcription + diarization backbone: full pyannote stack via API
- speech emotion / prosody: `emotion2vec+`-style model run on sentence-level segments derived from pyannote transcript output
- audio events: `YAMNet`-style model

Gemini can still interpret these signals semantically in Phase 3A, but should not be the primary detector for them.

## Artifact Sketches

### `canonical_timeline.json`

This should remain a single unified artifact in V3.1.

Reason:

- it is the main audio / text backbone for the run
- words, speaker turns, and sentence boundaries are tightly coupled and constantly consumed together
- keeping them in one file makes semantic replay, debugging, and versioning much simpler
- unlike render geometry, this artifact is not the heavy part of the visual pipeline

The planned split is therefore:

- keep `canonical_timeline.json` unified
- keep `shot_tracklet_index.json` lightweight
- store heavy bbox-over-time render data separately in `tracklet_geometry.json`

### `shot_tracklet_index.json`

This is the lightweight visual catalog for the run.

Its job is to answer:

- what shots exist
- which shot-local tracklets exist within each shot
- what lightweight metadata the UI and grounding flows need for each tracklet

It should contain:

- shot ranges
- tracklet IDs
- shot membership
- start / end timing
- visible duration / visible fraction
- median bbox center / size
- scale label
- representative thumbnails

It should not contain the heavy per-frame bbox-over-time data used by the renderer.

### `tracklet_geometry.json`

This is the render-oriented geometry artifact.

Its job is to store the heavy shot-local tracklet motion data needed for:

- follow crops
- reaction crops
- split layouts
- shot-by-shot render-plan compilation

It should contain:

- time-indexed bbox geometry for each tracklet
- any interpolation-ready position data needed by the render planner
- references or metadata needed to resolve geometry efficiently at render time

## Notes On Training Hooks

The future training / labeling loop is out of scope for MVP, but MVP should preserve the data required for it:

- all participation bindings should persist provenance
- all camera intent edits should persist separately from participation
- final accepted / rejected clip candidates should be retained as ranking feedback

This keeps the door open for a future proprietary supervised model path without making it an MVP dependency.

## User Interaction And Data-Write Flow

This section defines the MVP interaction model for the three grounding modes:

1. `speaker_id -> tracklet` within a shot
2. `word span -> tracklet` within a shot
3. `time span -> tracklet` within a shot

The core design rule is:

- these are three UI entry points into one normalized write model
- they must not become three separate backend truth systems

### Design Principles

- one canonical source of truth for user grounding writes
- materialized timelines are derived from user writes plus canonical media timelines
- explicit user writes always beat automated or default values
- speaker participation writes remain separate from camera intent writes
- speaker naming is orthogonal metadata, not a binding mechanism
- shot scope is the default scope for tracklet binding in MVP

### UX Surfaces

The grounding UI should expose the same underlying entities in all modes:

- speaker chips:
  - `Speaker_00`, `Speaker_01`, etc.
  - optional human-readable names like `Host`, `Guest`
- shot strip / shot selector
- visible bounding boxes / tracklet targets within the current shot
- diarized transcript with per-word highlighting
- time-range scrubber

### Core Write Model

V3.1 should store raw user actions in an append-only grounding log, then build the derived participation timeline from that log.

Recommended write artifacts:

- `speaker_registry.json`
- `grounding_events.jsonl`
- `participation_timeline.json`

#### `speaker_registry.json`

Video-scoped speaker metadata:

```json
{
  "video_id": "abc123",
  "speakers": [
    {
      "speaker_id": "Speaker_00",
      "display_name": "Host"
    },
    {
      "speaker_id": "Speaker_01",
      "display_name": "Guest"
    }
  ]
}
```

#### `grounding_events.jsonl`

`Append-only` means:

- when the user confirms a new edit, the system adds a new record
- it does not destructively overwrite prior records
- the current truth is rebuilt from the ordered history of confirmed edits

This is useful because:

- users can start with broad mappings and then refine smaller sections
- corrections do not require fragile in-place mutation
- future training analysis can inspect how labels were created and revised

The log stores user actions like:

```json
{
  "event_id": "ge_00001",
  "created_at": "2026-04-01T20:11:00Z",
  "video_id": "abc123",
  "clip_candidate_id": null,
  "event_type": "bind_speaker_to_tracklet",
  "scope": {
    "shot_id": "shot_012"
  },
  "payload": {
    "speaker_id": "Speaker_00",
    "tracklet_id": "shot_012_tr_03"
  }
}
```

Derived participation truth should not be edited directly.
It should be materialized from these events.

### Normalized Participation Assignment Shape

Regardless of how the user interacts, the system should normalize writes into assignment segments shaped like:

```json
{
  "start_ms": 1200,
  "end_ms": 4200,
  "speaker_ids": ["Speaker_00"],
  "tracklet_ids": ["shot_012_tr_03"],
  "binding_source": "speaker_id_map|word_span|time_span",
  "shot_id": "shot_012",
  "confidence": 1.0,
  "user_confirmed": true
}
```

Those normalized segments are then merged into `participation_timeline.json`.

### Grounding Mode 1: `speaker_id -> tracklet`

#### User Flow

1. user enters a shot
2. user clicks a visible bounding box / tracklet
3. user clicks a speaker chip such as `Speaker_00`
4. optional: user renames that speaker to `Host`
5. user confirms the shot-local mapping

#### Intended Semantics

This does not mean:

- "follow this person with the camera"
- "this person owns every exact word in the shot with word-perfect precision"

It means:

- "when this diarized speaker is the active speaker in this shot, this is the correct tracklet"

This is intentionally a coarse default mapping for normal non-overlap cases.

It is designed for common easy cases such as:

- a podcast shot where one diarized speaker clearly corresponds to one visible person
- an interview shot where the same person remains the obvious speaking subject throughout the shot

It is not intended to be the highest-precision truth for:

- overlapping speech
- fast interruptions
- reaction-only moments
- exact word ownership

Those cases should be refined with:

- `word span -> tracklet` for precise word ownership
- `time span -> tracklet` for interval-level participation corrections

#### Write Behavior

The system should append:

- one `bind_speaker_to_tracklet` event
- optional `rename_speaker` event

Suggested event shape:

```json
{
  "event_type": "bind_speaker_to_tracklet",
  "scope": {
    "shot_id": "shot_012"
  },
  "payload": {
    "speaker_id": "Speaker_00",
    "tracklet_id": "shot_012_tr_03"
  }
}
```

The materializer should then:

- look up diarized speaker turns for `Speaker_00` inside `shot_012`
- create normalized participation segments only for the overlapping portions of those turns

This keeps the write semantically tight and avoids incorrectly labeling the whole shot if the speaker is silent for part of it.

### Grounding Mode 2: `word span -> tracklet`

#### User Flow

1. user selects a shot
2. transcript view highlights words in sync with playback
3. user clicks or drags across a word span
4. user clicks a visible tracklet in that shot
5. user confirms the assignment

#### Intended Semantics

This is the highest-precision participation write.

It means:

- "these exact spoken words belong to this tracklet"

#### Write Behavior

The system should append a `bind_word_span_to_tracklet` event:

```json
{
  "event_type": "bind_word_span_to_tracklet",
  "scope": {
    "shot_id": "shot_012",
    "word_start_id": "w_000120",
    "word_end_id": "w_000156"
  },
  "payload": {
    "speaker_id": "Speaker_00",
    "tracklet_id": "shot_012_tr_03"
  }
}
```

The materializer should:

- resolve the selected words to `start_ms/end_ms`
- create an explicit normalized participation segment
- mark the segment as `binding_source=word_span`

This mode should take precedence over all broader mappings because it is the most explicit and precise.

### Grounding Mode 3: `time span -> tracklet`

#### User Flow

1. user selects a shot or clip candidate section
2. user drags across a time range on the scrubber
3. user clicks a visible tracklet
4. optional: user also associates one or more active `speaker_id`s if the UI knows them
5. user confirms

#### Intended Semantics

This means:

- "during this exact time period, this tracklet is the correct participant mapping"

This is useful when:

- the transcript is inconvenient to use
- the user is reasoning visually
- the user wants to capture a short reaction or ambiguous portion by time rather than text

Important:

- `time span -> tracklet` is an interval-level participation label
- it should not automatically be treated as exact word ownership unless the selected time span corresponds to a precise word-backed write
- if multiple people are assigned within the same time span, the system should treat that as multi-participant activity in the interval, not as proof of exact per-word speaker ownership

#### Write Behavior

The system should append a `bind_time_span_to_tracklet` event:

```json
{
  "event_type": "bind_time_span_to_tracklet",
  "scope": {
    "shot_id": "shot_012",
    "start_ms": 2500,
    "end_ms": 3500
  },
  "payload": {
    "speaker_ids": ["Speaker_00"],
    "tracklet_id": "shot_012_tr_03"
  }
}
```

The materializer should create an explicit normalized participation segment with `binding_source=time_span`.

### Speaker Renaming

Speaker renaming should be its own event type:

```json
{
  "event_type": "rename_speaker",
  "payload": {
    "speaker_id": "Speaker_00",
    "display_name": "Host"
  }
}
```

Rules:

- renaming is video-scoped for MVP
- renaming changes labels shown in the UI
- renaming does not itself create any participation mapping

### Conflict Resolution Rules

The materializer for `participation_timeline.json` should apply writes in precedence order.

#### Primary Rule: More Specific Scope Beats Less Specific Scope

The core precedence rule should be based on specificity, not on a hardcoded preference for one UI mode over another.

Highest to lowest specificity:

1. exact `word span -> tracklet`
2. bounded `time span -> tracklet`
3. shot-scoped `speaker_id -> tracklet`
4. no user binding

Rationale:

- word span is the narrowest and most transcript-anchored instruction
- time span is explicit but broader than a precise word selection
- shot speaker mapping is intentionally coarse and optimized for fast setup

This lets the user:

- start with broad shot-level mappings
- then refine smaller ranges without deleting the broader work

#### Precedence Within Same Write Type

- latest confirmed user write wins

This should let the user correct mistakes without destructive mutation of the event log.

#### Segment Splitting Behavior

If a narrower higher-precedence write overlaps a broader lower-precedence write, the materializer should split the broader segment around it rather than dropping it wholesale.

Example:

- shot-level mapping says `Speaker_00 -> tr_03` for the shot
- later word-span mapping corrects a short phrase to `tr_05`

The final participation timeline should become:

- before phrase: `tr_03`
- phrase: `tr_05`
- after phrase: `tr_03`

### Practical Interpretation

This precedence should be understood as:

- broad writes establish defaults
- narrower writes act as corrections or refinements
- later writes only win automatically when they operate at the same specificity level

### Unbinding / Correction

The user must be able to remove or override prior writes.

Recommended event:

```json
{
  "event_type": "clear_binding_range",
  "scope": {
    "shot_id": "shot_012",
    "start_ms": 2500,
    "end_ms": 3500
  }
}
```

This avoids destructive rewrites and keeps the event log useful for future training analysis.

### Materialized Participation Timeline

The final `participation_timeline.json` should be a merged non-overlapping timeline of user-confirmed speaker participation segments.

Suggested shape:

```json
[
  {
    "start_ms": 1200,
    "end_ms": 2100,
    "speaker_ids": ["Speaker_00"],
    "speaker_names": ["Host"],
    "tracklet_ids": ["shot_012_tr_03"],
    "binding_source": "speaker_id_map",
    "shot_id": "shot_012",
    "user_confirmed": true
  },
  {
    "start_ms": 2100,
    "end_ms": 2500,
    "speaker_ids": ["Speaker_00"],
    "speaker_names": ["Host"],
    "tracklet_ids": ["shot_012_tr_05"],
    "binding_source": "word_span",
    "shot_id": "shot_012",
    "user_confirmed": true
  }
]
```

### Relationship To Camera Intent

Participation writes and camera intent writes must remain separate.

Participation says:

- who is speaking / participating

Camera intent says:

- who to emphasize visually

So later, after candidate selection, the user may create camera intent segments such as:

```json
{
  "start_ms": 2100,
  "end_ms": 2500,
  "intent": "reaction",
  "primary_tracklet_id": "shot_012_tr_07"
}
```

even if the participation timeline for that same segment says:

- `Speaker_00 -> shot_012_tr_03`

That is valid and expected.

### Candidate-Scoped Writes For Finalist Grounding

For finalist-only routes, grounding events should optionally include:

- `clip_candidate_id`

This allows the system to distinguish:

- video-global early grounding
- finalist-local grounding

Recommended rule:

- finalist-local writes only materialize participation segments inside the candidate interval unless the user explicitly expands scope

### Implementation Notes

- all three grounding modes should write into one event stream
- the materializer should be deterministic and replayable
- derived timelines should be cheap to rebuild after every edit
- preview state can exist in the UI, but only confirmed edits should hit the event log
- camera intent should use the same event-sourcing pattern in a separate `camera_intent_events.jsonl` file

## Camera Intent Interaction And Data-Write Flow

This section defines the MVP interaction model for post-candidate camera direction:

- `follow`
- `reaction`
- `split`
- `wide`

The design goal is:

- let users direct the visual emphasis of finalist clips without mutating speaker participation truth

In this section:

- `camera intent` means the user's structured visual direction chosen through the UI
- it does not mean a chat command
- it is created through click / drag interactions on finalist clips

## Design Principles

- camera intent is a separate write stream from participation grounding
- camera intent is candidate-scoped by default
- camera intent writes are time-bounded, never assumed global unless the user explicitly expands scope
- shot changes should be respected by the compiler even if the user's intent segment crosses multiple shots
- the UI should feel simple, but the write model should be expressive enough to support rerendering without re-grounding or re-ranking

## Core Write Model

Recommended write artifacts:

- `camera_intent_events.jsonl`
- `camera_intent_timeline.json`
- `render_plan.json`

Camera intent should use the same append-only event log strategy as participation.

All camera intent writes should normalize to:

`primary_tracklet_id` and `secondary_tracklet_id` should be nullable depending on the intent type.

## Intent Types

### 1. `follow`

Meaning:

- keep the camera focused on this tracklet

Use cases:

- the speaking person should be the focus
- a single dominant on-screen subject should anchor the shot

Required fields:

- `primary_tracklet_id`

### 2. `reaction`

Meaning:

- the semantically important visual target is this tracklet's reaction, even if someone else is speaking

Use cases:

- podcast guest says something, but host reaction is the payoff
- disagreement moment where listener face sells the clip

Required fields:

- `primary_tracklet_id`

Important:

- `reaction` is not just a label alias for `follow`
- it is a semantic instruction that the renderer/compiler may use for UI copy, analytics, and future layout policy
- it is the main way for the user to intentionally show a non-speaking person's face during someone else's words

### 3. `split`

Meaning:

- present two tracklets together in an explicit split layout

Use cases:

- both participants matter
- disagreement / banter moments
- simultaneous reaction and speech both need to remain visible

Required fields:

- `primary_tracklet_id`
- `secondary_tracklet_id`

MVP rule:

- the order should be preserved in the write model so the UI can keep a stable top/bottom or left/right semantic ordering if needed

### 4. `wide`

Meaning:

- do not aggressively follow a single participant
- preserve wider framing

Use cases:

- both people are visible and spatial context matters
- shot is visually stable and crop-follow would reduce quality
- the user wants to preserve a shared composition

Required fields:

- no tracklet IDs required

## User Interaction Flows

The MVP camera-direction interaction model should feel like:

1. select when
2. select who
3. select how

Where:

- `when` = choose the interval by dragging words in the transcript or dragging a time range on the timeline
- `who` = click one or two visible bounding boxes / tracklets in the preview
- `how` = choose `Follow`, `Reaction`, `Split`, or `Wide`

This matters because the user may want to show someone who is not currently speaking.

Example:

- a guest speaks for a few words
- the host's reaction is more important than the speaker's face
- the user drags over those words in the transcript
- clicks the host's bounding box
- chooses `Reaction`

Result:

- participation timeline still says the guest is speaking
- camera intent says to emphasize the host for that short interval

### Primary Rule: More Specific Scope Beats Less Specific Scope

Use the same principle as participation:

- narrower time ranges override broader defaults
- later writes win within the same specificity level

### Intent Replacement

Camera intent is singular per instant in MVP.

That means:

- `follow`, `reaction`, `split`, and `wide` are mutually exclusive in the materialized timeline

If a user writes a narrower `reaction` inside a broader `follow`, the compiler should split the broader segment around it.

Example:

- `0-10s follow(tr_03)`
- `3-5s reaction(tr_07)`

Materialized result:

- `0-3s follow(tr_03)`
- `3-5s reaction(tr_07)`
- `5-10s follow(tr_03)`

Recommended event:

  }

## Candidate Scope And Expansion

For MVP, camera intent should default to `clip_candidate_id` scope.

Meaning:

- the user's direction is assumed to apply only to the current finalist clip
- it should not silently become a video-global preference

Optional later UX:

- "apply to entire source span"
- "reuse this intent on similar overlapping finalists"

But that is out of scope for the initial MVP write model.

## Default Camera Behavior From Participation

Participation provides the automatic camera baseline.
Custom camera choices override that baseline only where the user explicitly sets them.

Core rule:

- if custom camera intent exists for an interval, use it
- otherwise, derive the camera choice from the participation timeline

### MVP Default Policy Table

| Participation State | Default Camera Behavior | Why |
|---|---|---|
| One grounded active participant | `follow(primary tracklet)` | simplest and usually correct |
| One grounded active participant, but no valid visible target at part of the interval | `wide` for unresolved portion | avoid hallucinating a target |
| Two grounded active participants, both visible, geometry supports split | `split(tracklet_a, tracklet_b)` | both matter and can be shown clearly |
| Two grounded active participants, but split geometry is poor | `wide` | safer than bad split framing |
| More than two active participants | `wide` | MVP should avoid overcomplicated automatic layouts |
| No grounded participant | `wide` | no reliable follow target |
| One active speaker plus another interval-level active participant from overlap or coarse labeling | `split` if composition is clean, else `wide` | both matter, but avoid fake precision |
| One active speaker, one non-speaking but important reactor | `follow(active speaker)` by default | reaction importance is usually a human editorial choice unless explicitly overridden |

### Interpretation

Participation answers:

- who is active?

Default camera policy answers:

- if the user gives no extra direction, what is the safest and most reasonable shot choice?

Custom camera intent answers:

- what does the user actually want the viewer to look at?

So:

- participation drives automation
- camera intent drives editorial control

### Examples

#### Example 1: Simple speaking case

- participation:
  - `12.0s -> 16.0s Speaker_00 -> tr_03`
- no custom camera
- output:
  - `follow(tr_03)`

#### Example 2: Reaction override

- participation:
  - `12.0s -> 16.0s Speaker_00 -> tr_03`
- default:
  - `follow(tr_03)`
- user selects `13.1s -> 14.0s` and sets `reaction(tr_07)`

Final camera:

- `12.0s -> 13.1s follow(tr_03)`
- `13.1s -> 14.0s reaction(tr_07)`
- `14.0s -> 16.0s follow(tr_03)`

#### Example 3: Two-speaker overlap

- participation:
  - `20.0s -> 22.0s Speaker_00 -> tr_03`
  - `20.0s -> 22.0s Speaker_01 -> tr_06`
- default:
  - `split(tr_03, tr_06)` if visually clean
  - otherwise `wide`

Possible user overrides:

- `follow(tr_03)`
- `reaction(tr_06)`
- `wide`

#### Example 4: Ambiguous coarse interval

- participation:
  - `30.0s -> 31.5s tr_03 and tr_06 both active`
- default:
  - `split` or `wide`

If the user knows one face matters more, they can override with:

- `reaction(tr_06)`
- `follow(tr_03)`

## Shot Boundary Handling

### User Experience Rule

Users should be allowed to create one camera-choice segment that crosses shot boundaries.

They should not be forced to think in shot-sized chunks while editing.

### Compiler Rule

The compiler must split camera-choice segments at shot boundaries before producing the final render instructions.

Reason:

- tracklets are shot-local in V3.1
- a `follow` or `reaction` target cannot be assumed valid across a cut
- `split` layouts may need to rebind or degrade after a cut

Example:

- user writes `2000-7000ms follow(shot_012_tr_03)`
- shot changes at `5400ms`

Compiler output should become:

- `2000-5400ms follow(shot_012_tr_03)`
- `5400-7000ms unresolved_follow_after_cut`

Then the compiler should resolve the second segment using the rules below.

## Compiler Rules: Camera Intent -> Render Plan

The render compiler should read:

- `camera_intent_timeline.json`
- `participation_timeline.json`
- `canonical_timeline.json`
- `shot_tracklet_index.json`
- `tracklet_geometry.json`

and produce shot-by-shot render instructions.

`Shot-by-shot render instructions` means:

- the user's higher-level camera choices have been broken into segments that align with real shot boundaries
- each segment has been checked against the shot-local tracklets that actually exist there
- invalid targets after a cut have been degraded or marked for review
- the renderer receives concrete instructions it can execute directly

### Follow Resolution

For a `follow(tracklet)` segment:

- if the target tracklet exists in that shot segment, compile direct follow instructions
- if the segment crosses a cut and no valid corresponding target exists after the cut, do not invent a fake cross-shot identity
- degrade to:
  - `wide`, or
  - require user correction if this is a finalist preview step

MVP recommendation:

- preview degrades to `wide`
- UI should visually mark the segment as needing review

### Reaction Resolution

For a `reaction(tracklet)` segment:

- behave like `follow` geometrically
- preserve the semantic tag `reaction` in the render plan

Reason:

- later analytics / UX should know this was an intentional reaction-directed shot, not just a normal follow

### Split Resolution

For a `split(tracklet_a, tracklet_b)` segment:

- if both targets are valid and visible in the shot, compile split layout
- if one target becomes unavailable after a cut:
  - degrade to `follow` on the surviving valid target only if the user explicitly selected a fallback preference in future versions
  - otherwise degrade to `wide` and mark for review

MVP recommendation:

- degrade to `wide` and mark for review

### Wide Resolution

For a `wide` segment:

- preserve wide framing
- no tracklet validation required
- shot cuts do not invalidate the intent

- the materialized camera intent timeline

- the canonical tracklet / shot geometry

Rerendering should not require:

- re-running proposal extraction
- re-running graph construction
- re-running clip ranking

This is one of the key product advantages of the V3.1 architecture.

## Materializers And Compilers

V3.1 should introduce explicit derived-state builders.

### Materializers

Materializers take append-only user event logs and build current truth artifacts.

Required materializers:

- `materialize_participation.py`
- `materialize_camera_intent.py`

These should be:

- deterministic
- replayable
- cheap to rerun after every confirmed edit

### Compilers

Compilers turn user-facing abstractions into renderer-facing instructions.

Required compiler:

- `compile_camera_plan.py`

Inputs:

- participation timeline
- camera intent timeline
- shot boundaries
- tracklet geometry
- candidate clip boundaries

Output:

- `render_plan.json`

Important distinction:

- materializers produce current truth
- compilers produce executable instructions

## Service Boundaries

Recommended rule of thumb:

- `timeline/` owns canonical aligned evidence
- `grounding/` owns participation truth from user labeling
- `semantics/` owns proposal extraction and node creation
- `graph/` owns graph topology and storage
- `camera/` owns default camera policy plus user visual direction
- `render/` owns execution of final rendering
- `api/` owns transport and lifecycle only

This separation is important so that:

- rerender does not touch semantics
- participation relabeling does not require rerunning proposal extraction
- camera edits do not mutate graph truth

## Migration Sequence

Recommended implementation sequence:

1. create `backend/pipeline/v3_1/contracts.py` and `backend/pipeline/v3_1/artifacts.py`
2. build `timeline/` and produce run-scoped V3.1 artifacts from existing Phase 1 outputs
3. build `grounding/` event logs and materializers
4. build `semantics/` proposal extraction and node creation
5. build `graph/` on top of final semantic nodes
6. build `camera/` materialization plus default camera policy
7. adapt renderer through `render/` and `render_plan.json`
8. add API endpoints for grounding, camera choices, render-plan compilation, and rerender
9. wire `run_pipeline.py` or a separate runner into `backend/pipeline/v3_1/orchestrator.py`

- prefer thin wrapper scripts if CLI compatibility is needed
- do not store materialization logic inside API route handlers
- keep event logs and derived artifacts per-run and replayable
- treat `render_plan.json` as the contract between editorial choice and the renderer
