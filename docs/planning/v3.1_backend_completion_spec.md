# Clypt V3.1 Backend Completion Spec

**Status:** Draft for active implementation  
**Date:** 2026-04-04  
**Branch:** `v3.1-refactor`

## Purpose

This document defines the remaining backend work needed to turn the current refactored Phase 1-4 code into a real runnable V3.1 backend.

This is a completion spec, not a greenfield architecture doc.

It assumes:

- the current refactored Phase 1-4 modules under [backend/pipeline](/Users/rithvik/Clypt-V3/backend/pipeline) remain the canonical core
- we do **not** reintroduce the old legacy backend architecture
- we **do** selectively copy/adapt useful operational code from `main` where it saves time and does not drag legacy semantics back in

This spec covers:

1. what is already built
2. what is still missing
3. what needs to be copied/adapted from `main`
4. what needs to be built net-new
5. the runtime, worker, provider, env, and testing work required to make V3.1 actually runnable

## Core Build Principle

Do **not** waste time rewriting good operational code from scratch if `main` already has a useful implementation pattern.

But also:

Do **not** import old code wholesale.

The correct rule is:

- copy/adapt operational code from `main`
- rewrite contracts and orchestration around the new V3.1 pipeline
- keep all semantic/node/graph/candidate truth in the current refactored modules

## High-Level Missing Systems

There are six major missing systems:

1. provider integrations
2. runtime entrypoints
3. Phase 1 worker/service runtime
4. real embeddings and Gemini execution
5. artifact-producing Phase 1 sidecar execution
6. deployment + environment management

- pyannote
- Gemini generation
- Gemini embeddings or Vertex embeddings
- emotion2vec+
- YAMNet
- YouTube/media acquisition

### Proposed package boundary

Add a new package:


### Proposed runtime boundary

Add:

- `backend/phase1_runtime/`

Suggested layout:

```text

```

This mirrors the operational separation from `main`, but the implementation should be rewritten around V3.1 Phase 1 outputs.

### Phase 1 worker responsibilities

The new worker should:

1. accept a source URL
2. download the video/audio
3. produce a local normalized source video/audio workspace
4. run shot detection / tracking needed for `phase1_visual`
5. run pyannote API calls for diarization/transcription and optional identify
6. run emotion2vec+
7. run YAMNet
8. write normalized raw Phase 1 sidecar payloads
9. invoke the V3.1 pipeline Phase 1 transformation layer
10. optionally invoke the full Phase 1-4 orchestrator after Phase 1 succeeds

## System 4: Real Gemini And Embedding Execution

The current backend mostly validates/adapts Gemini responses supplied by the caller.

We need real runtime layers for:

- merge/classify neighborhood calls
- boundary reconciliation calls
- local semantic-edge calls
- long-range edge adjudication calls
- subgraph review calls
- pooled final-review calls
- node embeddings
- prompt embeddings

### Current placeholder behavior

Right now:

- [backend/pipeline/_embedding_utils.py](/Users/rithvik/Clypt-V3/backend/pipeline/_embedding_utils.py) provides deterministic local hash embeddings
- [backend/pipeline/semantics/prompts.py](/Users/rithvik/Clypt-V3/backend/pipeline/semantics/prompts.py) only covers Phase 2 prompt text
- Phase 3 and Phase 4 mostly validate structured results instead of making provider calls

### What to build

#### A. Gemini runtime adapter

Add:

- `backend/providers/gemini.py`

Responsibilities:

- client construction
- retry policy
- JSON schema / structured-output execution
- timeout handling
- logging and request IDs

#### B. Prompt builders

Add or extend:

- `backend/pipeline/semantics/prompts.py`
- new `backend/pipeline/graph/prompts.py`
- new `backend/pipeline/candidates/prompts.py`

Responsibilities:

- Phase 2 merge/classify prompts
- Phase 2 boundary reconciliation prompts
- Phase 3 local semantic-edge prompts
- Phase 3 long-range pair adjudication prompts
- Phase 4 subgraph review prompts
- Phase 4 pooled candidate review prompts

#### C. Embeddings provider

Add:

- `backend/providers/embeddings.py`

Responsibilities:

- live `semantic_embedding` generation for nodes from structured semantic payload
- live `multimodal_embedding` generation for nodes from local node video/audio spans
- live text embeddings for prompts and user queries
- a deterministic local fallback only for tests

Implementation note:

- the current deterministic embeddings should remain for tests
- production/runtime code should use a provider-backed path
- queries remain text embeddings
- first-pass retrieval should target `semantic_embedding`
- `multimodal_embedding` should be used as a second retrieval signal for reranking and long-range similarity enrichment

#### D. Provider-backed execution wrappers

Add small execution layers:

- `backend/pipeline/semantics/runtime.py`
- `backend/pipeline/graph/runtime.py`
- `backend/pipeline/candidates/runtime.py`

Responsibilities:

- batch construction
- prompt invocation
- structured response parsing
- debug artifact recording

Phase 1 currently expects these payloads to already exist:

- `pyannote_payload`
- `identify_payload`
- `emotion2vec_payload`
- `yamnet_payload`
- `phase1_visual`

We need code that actually produces them.

### 5A. Pyannote

Build:

- `backend/providers/pyannote.py`

Responsibilities:

- `diarize` request
- optional `identify` request
- request/response normalization
- retry behavior
- local caching of raw provider outputs

Runtime outputs to persist:

- `timeline/raw_pyannote_diarize.json`
- `timeline/raw_pyannote_identify.json` when used

### 5B. emotion2vec+

Build:

- `backend/providers/emotion2vec.py`

Responsibilities:

- model loading
- turn-level inference over canonical pyannote turns
- output normalization to the V3.1 emotion payload shape

Implementation note:

- this is a new net-new integration on this branch
- there is no equivalent old module to restore directly

### 5C. YAMNet

Build:

- `backend/providers/yamnet.py`

Responsibilities:

- audio loading
- patch/frame inference
- curated event-label filtering
- merged event span generation or raw detections before merge

Implementation note:

- this is also a net-new integration on this branch

### 5D. Visual Phase 1 extraction

We still need a V3.1-compatible `phase1_visual` producer.

This is the area most worth reusing from `main`.

What to reuse conceptually from `main` worker code:

- video decoding helpers
- shot-change detection path
- tracking path
- track geometry extraction

But the V3.1 output contract should be much narrower:

- `video_metadata`
- `shot_changes`
- `tracks`

Avoid reintroducing:

- active-speaker decisions
- face clustering as semantic truth
- speaker-binding artifacts
