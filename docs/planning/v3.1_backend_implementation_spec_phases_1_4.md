## Phase Map

### Phase 1: Timeline Foundation

Build the aligned run foundation:

- canonical transcript timeline
- speech-emotion timeline
- audio-event timeline
- shot tracklet index
- tracklet geometry

### Phase 2: Node Construction, Classification, And Embedding

Build merged semantic units from raw diarized turns:

- neighborhood batching with halos
- Gemini merge + classification
- boundary reconciliation for adjacent batch-edge nodes
- deterministic node embeddings

### Phase 3: Graph Construction

Build graph edges over merged semantic nodes:

- deterministic structural edges
- Gemini local semantic edges
- Gemini long-range callback/topic recurrence edges
- deterministic edge reconciliation

### Phase 4: Candidate Retrieval, Subgraph Selection, And Ranking

Build clip candidates from graph-local regions:

- duration-scaled meta prompts
- prompt embeddings
- seed retrieval
- local subgraph construction
- Gemini subgraph review
- candidate dedupe
- Gemini pooled final review

### Phase 2A: Turn neighborhoods

Raw diarized turns are the immutable ledger.

Neighborhood batching rules:

- contiguous target turn block
- halo turns on both sides
- halo is context only
- canonical nodes may only be built from contiguous whole turns

Each neighborhood payload carries:

- target/halo role per turn
- transcript text
- turn timings
- turn-level emotion evidence
- overlapping audio-event evidence

### Phase 2B: Merge + classify

One Gemini batch call per neighborhood does both:

- merged-unit boundary decisions
- final node classification

Rules:

- merges may only use contiguous whole turns
- target turns must be fully partitioned
- turns may not be skipped
- Gemini may not invent timestamps

### Phase 2C: Boundary reconciliation

A second smaller pass resolves disagreement at adjacent batch edges.

Current supported outcomes:

- `keep_both`
- `merge`

### Phase 2D: Node embeddings

Merged canonical nodes should ultimately receive two durable embeddings:

- `semantic_embedding`
- `multimodal_embedding`

Target implementation modules:

- [node_embeddings.py](/Users/rithvik/Clypt-V3/backend/pipeline/semantics/node_embeddings.py)
- [\_embedding_utils.py](/Users/rithvik/Clypt-V3/backend/pipeline/_embedding_utils.py)

Implementation target:

- `semantic_embedding`
  - canonical retrieval vector
  - embed a compact structured semantic payload:
    - summary
    - transcript text
    - node type
    - node flags
    - compact emotion summary
    - compact audio-event summary
- `multimodal_embedding`
  - second durable node vector
  - embed the node's local video/audio span in Gemini Embedding 2 space

Query rule:

- prompts and user queries remain text embeddings
- first-pass retrieval should target `semantic_embedding`
- `multimodal_embedding` is used as a secondary reranking / enrichment signal

Current implementation note:

- the current code still uses deterministic local test embeddings
- the provider-backed completion pass should upgrade this to the two-embedding runtime design above

### Phase 3A: Deterministic structural edges

Computed without Gemini:

- `next_turn`
- `prev_turn`
- `overlaps_with`

### Phase 3B: Gemini local semantic edges

Computed from local merged-node neighborhoods:

- `answers`
- `challenges`
- `contradicts`
- `supports`
- `elaborates`
- `setup_for`
- `payoff_of`
- `reaction_to`
- `escalates`

Rules:

- source node must be in the target block
- response may only emit supported semantic edge types
- rationale and confidence are optional but supported

### Phase 3C: Gemini long-range edges

Long-range adjudication is limited to:

- `callback_to`
- `topic_recurrence`

Candidate pairs are shortlisted by:

- node embeddings, with semantic similarity primary
- multimodal similarity as shortlist enrichment
- top-`k` nearest later-node retrieval

Semantic direction:

- shortlist search goes earlier -> later
- final edge usually points later -> earlier

### Phase 3D: Deterministic edge reconciliation

The pipeline, not Gemini, owns final edge reconciliation.

Rules:

- exact duplicates collapse
- support count is accumulated
- confidence may be averaged
- compatible multi-edge pairs may survive together
- incompatible edge types resolve by support count, confidence, then precedence

### Phase 4A: Meta prompts

Default prompt count is duration-scaled:

- under 5 minutes -> `2`
- 5 to 15 minutes -> `4`
- 15 to 25 minutes -> `6`
- 25 to 60 minutes -> `8`
- over 60 minutes -> `10`

### Phase 4B: Seed retrieval

Prompt embeddings are compared primarily against `semantic_embedding`.

Rules:

- top `5` seeds per prompt by cosine similarity
- `multimodal_embedding` may rerank or enrich the shortlist after semantic retrieval
- repeated seeds across prompts dedupe
- seed provenance is retained through `source_prompt_ids`
- one local time region may not dominate all seeds

### Phase 4C: Local subgraphs

Subgraphs are built deterministically around seed nodes using:

- weighted local expansion
- hop-depth limit
- branching-factor limit
- max duration
- max node count

Overlapping pre-review subgraphs dedupe when node overlap exceeds `0.70`.

### Phase 4D: Gemini subgraph review

Gemini reviews each local subgraph and may return:

- `reject_all`
- up to `3` good, meaningfully distinct clip candidates

Returned candidates include:

- node IDs
- start/end timestamps
- score
- rationale

Invalid subgraph output is normalized into a rejected subgraph result.

### Phase 4E: Candidate dedupe

Candidate dedupe is deterministic.

Near-duplicates collapse when either:

- node overlap exceeds `0.70`
- span IoU exceeds `0.70`

Tie-breaking is deterministic.

### Phase 4F: Pooled final review

All surviving candidates are sent to Gemini for the final keep/drop/rank pass.

Important rules:

- pooled review must not change boundaries
- pooled review may only keep, drop, and rank supplied candidates
- invalid pooled output fails hard
- missing or failed pooled review fails hard

There is intentionally no deterministic pooled-review fallback.
