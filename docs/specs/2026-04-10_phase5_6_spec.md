# Clypt V3.1 Spec: Phase 5-6 Grounding + Render

**Status:** Active (planned)  
**Date:** 2026-04-10  
**Scope:** Define the post-Phase-4 workflow for speaker participation grounding (Phase 5) and 9:16 render planning/output (Phase 6).

---

## 1. Locked Decisions

1. Phase 5 and Phase 6 remain **post-candidate** steps; no Phase 5-6 logic can run before Phase 4 candidate finalization.
2. Speaker participation truth and camera intent are separate artifacts and must never be conflated.
3. Global identity backbone remains diarized speaker IDs from Phase 1 timeline; visual identity remains shot-local tracklets.
4. Phase 5 writes are user-grounded and deterministic; no hidden auto-rewrites of user-confirmed mappings.
5. Phase 6 consumes canonical artifacts and grounded editorial intent to produce renderer-ready 9:16 instructions.
6. Rerendering must not require recomputing Phases 1-4.

---

## 2. Problem Statement

Phases 1-4 already generate strong ranked clip candidates with semantic graph evidence and (now) external signal augmentation. The pipeline still lacks:

- explicit, auditable speaker participation grounding on finalists,
- explicit editorial camera intent for each finalist interval,
- deterministic compilation into render-ready 9:16 instructions.

Without these, the system can rank moments but cannot reliably deliver production-ready outputs.

---

## 3. Non-Goals (Current Scope)

1. No full keyframe animation editor in MVP.
2. No global automatic face identity graph replacing diarization.
3. No training loop implementation in this phase (only data capture hooks).
4. No autonomous camera language generation that bypasses user grounding.

---

## 4. Phase 5: Participation Grounding + Camera Intent

## 4.1 Inputs

- Phase 4 `clip_candidates` finalists.
- Phase 1 timeline + visual artifacts:
  - `canonical_timeline.json`
  - `shot_tracklet_index.json`
  - `tracklet_geometry.json`

## 4.2 Grounding Model

Participation grounding supports three write paths that normalize to one truth model:

1. `speaker_id -> tracklet` (shot-scoped default mapping)
2. `word span -> tracklet` (highest precision)
3. `time span -> tracklet` (interval override)

All confirmed writes are event-sourced and materialized into a deterministic participation timeline.

## 4.3 Separation Rule

- **Participation** answers: who is speaking/participating.
- **Camera intent** answers: who to emphasize visually and how (`follow`, `reaction`, `split`).

This separation is mandatory for correctness and rerender flexibility.

## 4.4 Phase 5 Outputs

- `participation_timeline.json`
- `camera_intent_timeline.json`
- optional debug/provenance:
  - `grounding_events.jsonl`
  - `camera_intent_events.jsonl`
  - `speaker_registry.json`

---

## 5. Phase 6: Render Planning + 9:16 Output

## 5.1 Inputs

- Final shortlisted candidates from Phase 4.
- Materialized participation + camera intent from Phase 5.
- Shot/tracklet geometry from Phase 1.

## 5.2 Compiler Responsibilities

1. Split camera intent segments at shot boundaries.
2. Resolve valid tracklet targets per segment.
3. Mark unresolved segments for explicit review (no silent fallback).
4. Emit deterministic shot-by-shot render instructions.

## 5.3 Render Plan Contract

Primary artifact:

- `render_plan.json`

Each compiled segment should include, at minimum:

- `clip_candidate_id`
- `start_ms`, `end_ms`
- `shot_id`
- resolved layout mode
- resolved primary/secondary tracklet IDs
- `review_needed` flag
- semantic reason (`follow`, `reaction`, `split`, or unresolved reason)

---

## 6. Data Contracts (Phase 5-6 Additions)

## 6.1 Required Fields for Participation Segment

- `start_ms`, `end_ms`
- `speaker_ids`
- `tracklet_ids`
- `binding_source` (`speaker_id_map|word_span|time_span`)
- `shot_id`
- `user_confirmed`

## 6.2 Required Fields for Camera Intent Segment

- `start_ms`, `end_ms`
- `intent` (`follow|reaction|split`)
- `primary_tracklet_id`
- `secondary_tracklet_id` (nullable unless `split`)
- `clip_candidate_id`
- `user_confirmed`

---

## 7. Conflict Resolution Policy

1. More specific scope wins over less specific scope:
   - `word span` > `time span` > `speaker_id map`.
2. For same scope type, latest confirmed write wins.
3. Overlaps are resolved by segment splitting, not destructive erasure.
4. Camera intent is singular per instant in MVP; overlapping intent types are resolved deterministically.

---

## 8. Implementation Plan (File-Level)

1. Add Phase 5 package (planned): `backend/pipeline/grounding/`
   - contracts, event adapters, materializer.
2. Add camera package (planned): `backend/pipeline/camera/`
   - camera intent contracts, compiler, conflict resolver.
3. Add render package (planned): `backend/pipeline/render/`
   - render plan schema + renderer handoff.
4. Add persistence wiring for Phase 5-6 artifacts in repository layer.
5. Extend live runtime orchestration to invoke Phase 5-6 only after successful Phase 4.

---

## 9. Acceptance Criteria

1. A successful run can progress from Phase 4 finalists to persisted Phase 5 timelines.
2. `render_plan.json` is emitted with deterministic, shot-resolved segments.
3. Rerender can execute from stored artifacts without rerunning Phases 1-4.
4. Participation and camera intent edits are auditable and replayable.
5. Invalid/unresolved render segments are surfaced with explicit `review_needed` markers.
