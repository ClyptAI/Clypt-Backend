# V3.1 Phase 1 Known Issues

**Status:** Active troubleshooting notes  
**Branch:** `v3.1-refactor`

This document captures the current known issues with the deployed V3.1 Phase 1 runtime so we do not lose operational context between sessions.

## Current Issues

### Parallel runtime is not yet delivering the expected wall-clock speedup on longer videos

The V1 parallel runtime is functionally working, but longer inputs can still take longer than expected relative to the older serial path.

Current working hypothesis:

- the `audio` branch is the truly GPU-heavy branch
- the `visual` branch appears to be much more CPU-bound than expected on long videos
- as a result, the H200 is not being fully utilized during the full Phase 1 wall-clock

Observed symptoms from the deployed H200 worker:

- `audio` branch held roughly `23-24 GiB` of GPU memory during VibeVoice native inference
- `visual` branch held only roughly `2-3 GiB` of GPU memory while consuming very high CPU
- aggregate GPU utilization sat around `50-55%` during the long-running `joeroganflagrant.mp4` job

Implication:

- the current "parallel" pipeline overlaps branches, but not as efficiently as intended
- the likely bottleneck is visual-side frame decode, batching, post-processing, tracking, or another CPU-heavy stage rather than raw detector inference

### Branch progress observability is still weak

Branch log quality is uneven:

- `yamnet` logs are informative enough
- `visual` logs show startup / RF-DETR warmup but little steady-state progress
- `audio` branch log can remain effectively empty even while the nested VibeVoice subprocess is active

Implication:

- it is too hard to distinguish "slow but healthy" from "stalled" without checking `nvidia-smi`, `ps`, or branch status files directly

### RF-DETR warmup / compile cost is still noticeable

The visual branch still pays a real first-run warmup and tracing cost on fresh worker boots.

Observed contributing factor:

- warning about missing `PYTORCH_KERNEL_CACHE_PATH` or `HOME`, which disables kernel caching

Implication:

- cold-start visual latency is higher than it should be
- repeated jobs on the same worker may behave better than first-run jobs

### YouTube ingestion remains operationally brittle

Even though the runtime supports `yt-dlp`, direct YouTube ingestion is still less reliable than testing from local MP4s copied onto the droplet.

Current testing preference:

- keep source videos in the local `videos/` directory
- SCP or rsync them onto the worker
- submit jobs using `source_path`

### YAMNet must remain CPU-isolated in the V1 parallel runtime

This is now fixed in code, but it remains an important operational invariant.

Previous failure mode:

- TensorFlow / YAMNet accidentally saw the GPU
- YAMNet allocated large GPU memory
- VibeVoice native then OOMed during model load

Current mitigation:

- YAMNet branch explicitly strips GPU visibility from its subprocess environment
- TensorFlow provider also disables visible GPUs in CPU mode

## What Is Confirmed Working

### End-to-end Phase 1 can succeed on the deployed H200 worker

Confirmed successful run:

- source: `mrbeastflagrant.mp4`
- result: full Phase 1 success
- outputs included:
  - `phase1_visual`
  - `diarization_payload`
  - `emotion2vec_payload`
  - `yamnet_payload`

### V1 branch model is functionally sound

Confirmed:

- `visual` and `audio` can run concurrently
- `yamnet` can run separately on CPU
- branch join / result wiring works
- service + worker shared DB/log path fixes are working

## Next Investigation Areas

1. Profile the `visual` branch for CPU-heavy hotspots on long videos.
2. Add branch-level heartbeat/progress logging for both `visual` and `audio`.
3. Improve RF-DETR kernel cache behavior on the worker.
4. Benchmark long-video serial vs parallel runs on the same source to measure the real overlap gain or loss.
