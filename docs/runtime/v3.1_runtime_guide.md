### YouTube provider env

- `CLYPT_YOUTUBE_COOKIES_FILE` — path to a Netscape-format cookies.txt file; required on headless servers for bot-detection bypass
- `CLYPT_YOUTUBE_JS_RUNTIMES` — JS runtime for yt-dlp's EJS n-challenge solver (e.g. `bun:/root/.bun/bin/bun`); required alongside cookies — without this, yt-dlp fails with "Requested format is not available" even when cookies are valid. **Use `bun` not `node`** — yt-dlp only bundles EJS solver scripts for bun and deno, not node

### Visual pipeline env

- `CLYPT_PHASE1_VISUAL_BACKEND` — detector inference backend (default: `pytorch_cuda_fp16`, alt: `tensorrt_fp16`)
- `CLYPT_PHASE1_VISUAL_BATCH_SIZE` — frames per detector batch (default: `4`)
- `CLYPT_PHASE1_VISUAL_THRESHOLD` — detection confidence threshold (default: `0.35`)
- `CLYPT_PHASE1_VISUAL_SHAPE` — detector input resolution (default: `640`); **must be 640** — RF-DETR's backbone requires inputs divisible by 32 and uses patch size 16; `560` triggers an AssertionError at runtime
- `CLYPT_PHASE1_VISUAL_TRACKER` — tracker backend (default: `bytetrack`)
- `CLYPT_PHASE1_VISUAL_TRACKER_BUFFER` — lost track buffer frames (default: `30`)
- `CLYPT_PHASE1_VISUAL_TRACKER_MATCH_THRESH` — ByteTrack `track_activation_threshold` (default: `0.7`); this is ByteTrack's upstream default — do not raise it without testing, as it delays track activation
- `CLYPT_PHASE1_VISUAL_DECODE` — frame decode backend (default: `cpu`)
- `CLYPT_PHASE1_VISUAL_TRT_ENGINE_DIR` — directory for cached TensorRT engines (default: `backend/outputs/tensorrt_engines`)

## Live provider clients

### VibeVoice ASR

- [backend/providers/vibevoice.py](/Users/rithvik/Clypt-V3/backend/providers/vibevoice.py)
- Native worker (second venv only): [backend/runtime/vibevoice_native_worker.py](/Users/rithvik/Clypt-V3/backend/runtime/vibevoice_native_worker.py)

Implements diarization + transcription in a single pass, replacing Pyannote cloud.

**Default (`VIBEVOICE_BACKEND=native`):** spawns `VIBEVOICE_NATIVE_VENV_PYTHON -m backend.runtime.vibevoice_native_worker` with a JSON job on stdin. That interpreter loads `microsoft/VibeVoice-ASR` (~7B) via the official `vibevoice` package — same dependency line as Microsoft’s Gradio demo, isolated from the main worker’s `transformers>=5` (RF-DETR).

**Optional (`VIBEVOICE_BACKEND=hf`):** in-process Hugging Face `microsoft/VibeVoice-ASR-HF` (~9B) using `VibeVoiceAsrForConditionalGeneration`; uses the **same** primary venv as the rest of Phase 1.

Features:

- `run(audio_path, context_info=None) -> list[dict]` — returns `[{Start, End, Speaker, Content}, ...]`
- Greedy decoding defaults (`do_sample=False`, `num_beams=1`, etc.) unless overridden via env
- HF path: optional custom diarization-focused system prompt via `VIBEVOICE_SYSTEM_PROMPT`
- `context_info` / `VIBEVOICE_HOTWORDS_CONTEXT` — hotwords/context string (comma-separated in `.env`) injected into the request
- Supports long audio; multilingual
- Speaker IDs are integers (0, 1, 2...) — normalized to `SPEAKER_0`, `SPEAKER_1` by `vibevoice_merge.py`
- Audio duration / RTF logging uses `torchaudio.info(...)` only. We intentionally removed the `wave` fallback after it reported bogus `2**30 - 1` frame counts on `joeroganxflagrant.wav` and logged the clip as ~6.8 hours instead of ~13.1 minutes
- HF fallback path downmixes stereo input to mono before inference; this avoids `Audio should be mono` errors on dual-channel WAVs such as `joeroganxflagrant.wav`

**Model download:** first run pulls weights from Hugging Face into `~/.cache/huggingface/` (size depends on 7B vs 9B model).

### Forced alignment (NeMo Forced Aligner)

- [backend/providers/forced_aligner.py](/Users/rithvik/Clypt-V3/backend/providers/forced_aligner.py)

Adds per-word timestamps to VibeVoice's turn-level output using NVIDIA NeMo Forced Aligner (NFA).

- `run(audio_path, turns) -> list[dict]` — returns `[{word_id, text, start_ms, end_ms, speaker_id}, ...]`
- Slices each turn's audio segment, runs Viterbi CTC forced alignment, returns absolute word timestamps
- Uses `stt_en_fastconformer_hybrid_large_pc` (FastConformer hybrid model in CTC mode, ~1GB, auto-downloaded from NGC on first run)
- Degrades gracefully: if `nemo_toolkit[asr]` is not installed, logs a warning and returns empty `[]` — the rest of the pipeline uses token-split fallback timing in `vibevoice_merge.py`

### Parent coordinator and branch workers

- [backend/phase1_runtime/extract.py](/Users/rithvik/Clypt-V3/backend/phase1_runtime/extract.py)
- [backend/phase1_runtime/coordinator.py](/Users/rithvik/Clypt-V3/backend/phase1_runtime/coordinator.py)
- [backend/runtime/run_phase1_branch.py](/Users/rithvik/Clypt-V3/backend/runtime/run_phase1_branch.py)

The live default is now a parent-coordinated Phase 1 runtime. `factory.py` enables it when `CLYPT_PHASE1_PARALLEL_ENABLED=1`, validates the V1 contract (`CLYPT_PHASE1_PARALLEL_GPU_BRANCH_LIMIT=2`, `CLYPT_PHASE1_YAMNET_BRANCH_DEVICE=cpu`), and passes the configured branch timeout/poll interval into the coordinator.

The coordinator prepares one run workspace, launches three isolated branch workers as child processes, waits for their result artifacts, and joins them back into `Phase1SidecarOutputs`.

Implemented branch model:

1. `visual` — RF-DETR Large + ByteTrack
2. `audio` — VibeVoice-ASR, NeMo Forced Aligner, `vibevoice_merge`, `emotion2vec+`
3. `yamnet` — YAMNet event classification

V1 scheduling and isolation rules:

- `visual` and `audio` are the two GPU-heavy branches and may overlap
- `yamnet` is intentionally CPU-only in V1, even on GPU hosts
- child stdout/stderr is redirected into per-branch log files
- if one branch hard-fails or times out, the coordinator terminates siblings, writes a run-level `branch_summary.json`, and raises a branch-aware failure back to the worker

Why VibeVoice remains a nested subprocess:

- the validated production path still depends on a second native venv with the official `vibevoice` package and its pinned dependency stack
- keeping VibeVoice nested inside the `audio` branch isolates that environment from the main Phase 1 worker env, which also owns RF-DETR and the rest of the runtime
- this preserves the known-good H100/H200 path without forcing the entire worker onto VibeVoice’s dependency stack

The `Phase1SidecarOutputs.diarization_payload` field (formerly `pyannote_payload`) contains the merged `{words, turns}` dict from `vibevoice_merge.py`. The `identify_payload` field has been removed — VibeVoice does not support external voiceprint identification.

### Filesystem layout and observability

Each run workspace lives under `CLYPT_PHASE1_WORK_ROOT/<run_id>/`.

Important paths:

- `source_video.mp4`
- `source_audio.wav`
- `metadata/branch_summary.json`
- `branches/visual/request.json`
- `branches/visual/status.json`
- `branches/visual/result.json`
- `branches/visual/branch.log`
- same contract for `branches/audio/` and `branches/yamnet/`

The branch summary captures:

- the failed branch, if any
- request/status/result/log paths per branch
- per-branch status snapshots
- termination details for the failed branch and any siblings the coordinator shut down

`run_phase1_branch.py` writes:

- `status.json` with `queued -> running -> succeeded|failed`
- `result.json` on success, or a failure envelope with `branch` and `branch_log_path`
- branch logs via the parent process redirecting child stdout/stderr into `branch.log`

### Visual extraction

The visual extraction stack has been refactored to use RF-DETR Large + ByteTrack:

- **Detector:** `RFDETRLarge` from the `rfdetr` library
- **Tracker:** `ByteTrackTracker` from Roboflow `trackers`
- **Decode:** CPU-based frame decoding via OpenCV

The old Ultralytics/YOLO-based detector path has been removed.

Architecture:

1. `visual.py` — orchestration: metadata probe, shot detection, detector factory, pipeline coordination, artifact post-processing
2. `rfdetr_detector.py` — PyTorch CUDA RF-DETR model loading, FP16 optimization, person-only detection
3. `tensorrt_detector.py` — ONNX export, TensorRT engine build, native TRT FP16 inference, person-only detection
4. `frame_decode.py` — CPU video frame decoding, batch assembly
5. `tracker_runtime.py` — ByteTrackTracker lifecycle, sequential frame updates, track row emission
6. `visual_config.py` — env-driven configuration for detector/tracker/decode/TRT settings

The detector factory in `visual.py` (`_make_detector`) selects between `RFDETRPersonDetector` (PyTorch CUDA) and `TensorRTDetector` (native TRT) based on `config.use_tensorrt`.

Key properties:

- CUDA is required for detection — no silent CPU fallback
- FP16 inference is the default
- Two detector backends:
  - `pytorch_cuda_fp16` — PyTorch with FP16 and torch.compile (default, works on any CUDA GPU)
  - `tensorrt_fp16` — native TensorRT engine with FP16 (fastest, requires TensorRT install)
- TensorRT path auto-exports ONNX and converts to `.engine` on first run, then caches
- Engines are GPU-specific: rebuild when switching GPU families
- detection is batched, tracking is sequential in frame order
- person class only (COCO class **1** — RF-DETR uses class 0 for background; class 1 is person)

Visual extraction emits:

- `video_metadata`
- `shot_changes` (via ffmpeg scene-cut detection)
- `tracks` (shot-local, with normalized geometry and bbox_norm_xywh)
- `person_detections` (derived from shot-split tracks)
- `tracking_metrics` (with RF-DETR runtime labels, latency, and throughput)

Current intentional omissions in the visual payload:

- `face_detections`
- `visual_identities`
- `mask_stability_signals`

Those remain empty in the current Phase 1 runtime because the V3.1 Phase 1-4 build does not revive the old face identity / LR-ASD stack.
