# V3.1 Runtime Guide

**Status:** Active backend runtime reference
**Branch:** `v3.1-refactor-vLLM`

This document explains the current live runtime/provider surface for the V3.1 Phase 1 worker using the vLLM VibeVoice sidecar.

## Current Working Setup

- Phase 1 visual extraction and ASR run **concurrently** ‚Äî visual uses GPU, ASR is an HTTP call to the local vLLM service, so they don't contend
- As soon as ASR completes, NFA ‚Üí emotion2vec+ ‚Üí YAMNet start immediately (RF-DETR still running); serial with each other, concurrent with visual
- Audio chain artifacts ready ~230s before RF-DETR finishes on a 13-min clip ‚Äî Phases 2‚Äì4 can begin earlier
- **ASR:** `VibeVoiceVLLMProvider` ‚Äî sends audio to `clypt-vllm-vibevoice.service` over localhost HTTP
- **Visual:** RF-DETR Small + ByteTrack (`pytorch_cuda_fp16` default, `tensorrt_fp16` available)
- **Model name:** `vibevoice` (NOT `microsoft/VibeVoice-ASR`)


## Live Provider Clients

### VibeVoice ASR ‚Äî vLLM backend

- [backend/providers/vibevoice_vllm.py](/Users/rithvik/Clypt-V3/backend/providers/vibevoice_vllm.py)

Implements diarization + transcription by sending audio to the persistent `clypt-vllm-vibevoice.service` Docker container.

**Key behaviors:**

- `load()` ‚Äî health-checks the vLLM service via `GET /health`; raises immediately if the service is unreachable
- `run(audio_path, context_info=None) -> list[dict]`
  - `output_mode=turns`: returns `[{Start, End, Speaker, Content}, ...]`
  - `output_mode=words`: returns `[{start_ms, end_ms, speaker_id, word}, ...]`
- **MP4/video extraction:** detects video extensions (`.mp4`, `.mov`, `.mkv`, etc.) and runs `ffmpeg -vn -acodec libmp3lame -q:a 2` to extract a temporary MP3 before sending. The temp file is cleaned up in a `finally` block. This is required ‚Äî sending a raw 68MB MP4 as base64 causes response truncation.
- **Duration probing:** uses `ffprobe` (not `torchaudio`) to get the audio duration. The duration is embedded in the vLLM prompt as `"This is a {duration:.2f} seconds audio..."` ‚Äî an incorrect duration degrades transcription quality.
- **Prompt format:** selects schema by `VIBEVOICE_OUTPUT_MODE`:
  ```
  turns mode: ... keys: Start time, End time, Speaker ID, Content
  words mode: ... keys: start_ms, end_ms, speaker_id, word
  ```
  Accompanied by a system message: `"You are a helpful assistant that transcribes audio input into text output in JSON format."`
- **Streaming:** uses `stream=True` in the API request and reads SSE chunks. Non-streaming causes response truncation at ~13K characters for long audio.
- **`supports_concurrent_visual = True`:** class attribute that signals `extract.py` to run visual extraction and ASR in parallel via `ThreadPoolExecutor`. The vLLM HTTP call does not use the GPU, so there is no CUDA contention.
- `teardown()` ‚Äî no-op; the service is persistent

### Forced Alignment (NeMo Forced Aligner)

- [backend/providers/forced_aligner.py](/Users/rithvik/Clypt-V3/backend/providers/forced_aligner.py)

Adds per-word timestamps to VibeVoice turn-level output using NVIDIA NeMo's CTC forced aligner (`VIBEVOICE_OUTPUT_MODE=turns` path).

- `run(audio_path, turns) -> list[dict]` ‚Äî returns `[{word_id, text, start_ms, end_ms, speaker_id}, ...]`
- Uses `stt_en_fastconformer_hybrid_large_pc` (~1GB, auto-downloads from NGC on first run)
- Degrades gracefully: if `nemo_toolkit[asr]` is not installed, logs a warning and returns `[]`

### Vertex AI

- [backend/providers/vertex.py](/Users/rithvik/Clypt-V3/backend/providers/vertex.py)

#### `VertexGeminiClient`

- `generate_json(prompt, model, temperature, response_schema, max_output_tokens)` ‚Äî Gemini Pro (or Flash) via Vertex
- Uses the **global** Vertex endpoint (`VERTEX_GEMINI_LOCATION=global`) for Priority PayGo support
- Passes **Priority PayGo headers** on the generation client: `X-Vertex-AI-LLM-Shared-Request-Type: priority` + `X-Vertex-AI-LLM-Request-Type: shared` ‚Äî eliminates DSQ tail latency (~4‚Äì6 min delay on calls 4‚Äì5 in a concurrent burst)
- Disables thinking tokens via `ThinkingConfig(thinking_budget=0)` on every call ‚Äî `ThinkingLevel.HIGH` was causing 90‚Äì390s per batch call and is fully removed
- Passes `response_schema` (JSON Schema dict) to `GenerateContentConfig.response_schema` for constrained decoding ‚Äî faster generation, no hallucinated fields, no parse failures
- Passes `max_output_tokens` cap per call type ‚Äî reduces per-slot token reservation in the Vertex AI scheduler

Per-phase schemas and token caps:
| Phase | Call | Schema | max_output_tokens |
|---|---|---|---|
| Phase 2 | Merge/classify | `MERGE_AND_CLASSIFY_SCHEMA` | 2048 |
| Phase 2 | Boundary reconciliation | `BOUNDARY_RECONCILIATION_SCHEMA` | 1024 |
| Phase 3 | Local semantic edges | `LOCAL_SEMANTIC_EDGE_SCHEMA` | 2048 |
| Phase 3 | Long-range edges | `LONG_RANGE_EDGE_SCHEMA` | 1024 |
| Phase 4 | Meta-prompt generation | `META_PROMPT_GENERATION_SCHEMA` | 1024 |
| Phase 4 | Subgraph review | `SUBGRAPH_REVIEW_SCHEMA` | 2048 |
| Phase 4 | Pool review | `POOL_REVIEW_SCHEMA` | 2048 |

#### `VertexEmbeddingClient`

- `embed_texts(text_list)` ‚Äî Gemini Embedding 2 via Vertex (`us-central1` ‚Äî does not support Priority PayGo)
- `embed_media_uris(uri_list)` ‚Äî multimodal GCS URI embeddings

**Critical API constraint:** `embed_content` treats a list of text strings as a **single** multimodal document and returns 1 embedding, not N. Similarly, it accepts at most 1 video part per call. Both methods use a `ThreadPoolExecutor` to dispatch one `embed_content` call per item in parallel ‚Äî never batch.

### Sidecar orchestration

- [backend/phase1_runtime/extract.py](/Users/rithvik/Clypt-V3/backend/phase1_runtime/extract.py)

Execution model (vLLM concurrent):
```
visual extraction ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê
                    ThreadPoolExecutor(max_workers=3)‚îú‚îÄ‚îÄ both done
ASR (HTTP) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò                               ‚îÇ
    ‚Üì asr_future.result() ‚Äî starts immediately       ‚îÇ
NFA ‚Üí emotion2vec+ ‚Üí YAMNet ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
    (serial with each other; concurrent with RF-DETR)
```

The code calls `asr_future.result()` directly (blocking until ASR completes), immediately submits `_run_audio_chain()` as a third worker, joins the audio chain, fires the `on_audio_chain_complete` callback (while RF-DETR still runs), then joins `visual_future`.

**Important:** Do NOT use `as_completed([visual_future, asr_future])` here. That pattern appeared to work but actually caused the callback to wait for both futures ‚Äî the loop only exited after RF-DETR finished, firing the callback ~33s late. See deployment doc for full explanation.

Output: canonical `Phase1SidecarOutputs` fed into the Phase 2‚Äì4 pipeline.

### Visual extraction

- [backend/phase1_runtime/visual.py](/Users/rithvik/Clypt-V3/backend/phase1_runtime/visual.py)

Stack: `RFDETRSmall` (rfdetr library) + `ByteTrackTracker` (Roboflow trackers) + CPU frame decoding via OpenCV.

Key rules:
- CUDA required ‚Äî no CPU fallback
- Person class: COCO class **1** (class 0 is background in RF-DETR output)
- `CLYPT_PHASE1_VISUAL_SHAPE` must be `640` ‚Äî backbone requires inputs divisible by 32 with patch size 16
- `ByteTrackTracker` must be initialized with explicit `lost_track_buffer`, `frame_rate`, `track_activation_threshold` ‚Äî default constructor ignores config
- Frame rate passed to tracker must match actual video FPS (detected via `cv2.VideoCapture`)
- Filtering `sv.Detections` by boolean mask requires reconstructing detections without `det.data` to avoid `IndexError`

Visual output: `video_metadata`, `shot_changes`, `tracks`, `person_detections`, `tracking_metrics`

### Phase 4: Dynamic Meta-Prompt Generation

Phase 4 retrieval is driven by **video-specific meta-prompts** generated by Gemini Flash (`VERTEX_FLASH_MODEL`). The generator reads Phase 2 node summaries and produces targeted retrieval queries scaled to video duration:

| Duration | Prompt count |
|---|---|
| < 5 min | 2 |
| 5‚Äì15 min | 4 |
| 15‚Äì25 min | 6 |
| 25‚Äì60 min | 8 |
| > 60 min | 10 |

Implementation: `generate_meta_prompts_live()` in `backend/pipeline/candidates/runtime.py`. Uses `META_PROMPT_GENERATION_SCHEMA` for constrained decoding. **No fallback to static prompts** ‚Äî if Gemini returns an empty list, the function raises `ValueError`. Debug output saved to `candidates/meta_prompts_debug.json`.

## Service Infrastructure

Three runtime modes sharing the same extraction logic:

- **CLI** ‚Äî `backend/runtime/run_phase1.py`: single job, local execution
- **FastAPI service** ‚Äî `backend/phase1_runtime/app.py` + `backend/runtime/run_phase1_service.py`: HTTP job submission
- **Worker loop** ‚Äî `backend/phase1_runtime/worker.py` + `backend/runtime/run_phase1_worker.py`: polls SQLite for pending jobs

Job state: SQLite via `backend/phase1_runtime/state_store.py`

Additional tooling:
- `scripts/do_phase1/run_remote_job.py` / `run_remote_job.sh` ‚Äî submit jobs to the droplet worker from local
