### YouTube provider env

- `CLYPT_YOUTUBE_COOKIES_FILE` — path to a Netscape-format cookies.txt file; required on headless servers for bot-detection bypass
- `CLYPT_YOUTUBE_JS_RUNTIMES` — JS runtime for yt-dlp's EJS n-challenge solver (e.g. `node:/usr/bin/node`); required alongside cookies — without this, yt-dlp fails with "Requested format is not available" even when cookies are valid

### Visual pipeline env

- `CLYPT_PHASE1_VISUAL_BACKEND` — detector inference backend (default: `pytorch_cuda_fp16`, alt: `tensorrt_fp16`)
- `CLYPT_PHASE1_VISUAL_BATCH_SIZE` — frames per detector batch (default: `4`)
- `CLYPT_PHASE1_VISUAL_THRESHOLD` — detection confidence threshold (default: `0.35`)
- `CLYPT_PHASE1_VISUAL_SHAPE` — detector input resolution (default: `640`); **must be 640** — RF-DETR's backbone requires inputs divisible by 32 and uses patch size 16; `560` triggers an AssertionError at runtime
- `CLYPT_PHASE1_VISUAL_TRACKER` — tracker backend (default: `bytetrack`)
- `CLYPT_PHASE1_VISUAL_TRACKER_BUFFER` — lost track buffer frames (default: `30`)
- `CLYPT_PHASE1_VISUAL_TRACKER_MATCH_THRESH` — ByteTrack `track_activation_threshold` (default: `0.7`); this is ByteTrack's upstream default — do not raise it without testing, as it delays track activation
- `CLYPT_PHASE1_VISUAL_DECODE` — frame decode backend (default: `cpu`)
- `CLYPT_PHASE1_VISUAL_TRT_ENGINE_DIR` — directory for cached TensorRT engines (default: `backend/outputs/tensorrt_engines`)

- [backend/providers/pyannote.py](/Users/rithvik/Clypt-V3/backend/providers/pyannote.py)

- `run_diarize(media_url=...)`

Current mode:

**Important:** Pyannote expects an **audio URL**, not a video URL. The runner uploads the extracted 16kHz mono WAV to GCS, generates a V4 signed HTTPS URL for it, and passes that to the diarize call. Sending a video URL works (Pyannote extracts audio server-side) but is wasteful — the WAV is already extracted and is much smaller.

### Visual extraction

The visual extraction stack has been refactored to use RF-DETR Large + ByteTrack:

- **Detector:** `RFDETRLarge` from the `rfdetr` library
- **Tracker:** `ByteTrackTracker` from Roboflow `trackers`
- **Decode:** CPU-based frame decoding via OpenCV

The old Ultralytics/YOLO-based detector path has been removed.

Architecture:

1. `visual.py` — orchestration: metadata probe, shot detection, detector factory, pipeline coordination, artifact post-processing
2. `rfdetr_detector.py` — PyTorch CUDA RF-DETR model loading, FP16 optimization, person-only detection
3. `tensorrt_detector.py` — ONNX export, TensorRT engine build, native TRT FP16 inference, person-only detection
4. `frame_decode.py` — CPU video frame decoding, batch assembly
5. `tracker_runtime.py` — ByteTrackTracker lifecycle, sequential frame updates, track row emission
6. `visual_config.py` — env-driven configuration for detector/tracker/decode/TRT settings

The detector factory in `visual.py` (`_make_detector`) selects between `RFDETRPersonDetector` (PyTorch CUDA) and `TensorRTDetector` (native TRT) based on `config.use_tensorrt`.

Key properties:

- CUDA is required for detection — no silent CPU fallback
- FP16 inference is the default
- Two detector backends:
  - `pytorch_cuda_fp16` — PyTorch with FP16 and torch.compile (default, works on any CUDA GPU)
  - `tensorrt_fp16` — native TensorRT engine with FP16 (fastest, requires TensorRT install)
- TensorRT path auto-exports ONNX and converts to `.engine` on first run, then caches
- Engines are GPU-specific: rebuild when switching GPU families
- detection is batched, tracking is sequential in frame order
- person class only (COCO class **1** — RF-DETR uses class 0 for background; class 1 is person)

Visual extraction emits:

- `video_metadata`
- `shot_changes` (via ffmpeg scene-cut detection)
- `tracks` (shot-local, with normalized geometry and bbox_norm_xywh)
- `person_detections` (derived from shot-split tracks)
- `tracking_metrics` (with RF-DETR runtime labels, latency, and throughput)

Current intentional omissions in the visual payload:

- `face_detections`
- `visual_identities`
- `mask_stability_signals`

Those remain empty in the current Phase 1 runtime because the V3.1 Phase 1-4 build does not revive the old face identity / LR-ASD stack.
